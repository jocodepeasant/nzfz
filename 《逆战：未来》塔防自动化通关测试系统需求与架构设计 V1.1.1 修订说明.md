# 1. 修订说明

本文档是 V1.1 的修订补充说明，用于澄清 V1.1 中的若干协议边界、实现约束和执行器兼容性问题。

V1.1 已经确定以下核心设计：

```text
1. 地图配置器与执行器分离
2. 两者通过脚本包通信
3. 脚本包包含 script.json 和资源文件
4. schema_version 使用 1.1.0
5. 新增 compatibility 执行器能力声明
6. 新增 package 资源包定义
7. 所有 ratio 坐标基于 GameViewportRect
8. coordinate_mode 使用 viewport_ratio_after_region_enter
9. recognition 增加 rois/templates/color_rules/multi_frame
10. traps 增加 recognition 引用
11. conditions 使用 operator/items 标准结构
12. retry 使用 max_attempts
13. wave action 支持 action_id/enabled
14. 支持 wait/log 标准动作
15. 地图编辑器必须实现三层校验
```

V1.1.1 在此基础上重点修订：

```text
1. 明确配置器内部坐标、content_rect 与导出 GameViewportRect ratio 的关系
2. 补充多楼层协议字段
3. 明确 place_trap 自动 ensure_region 语义
4. 新增 capability 推导和映射规则
5. 区分 basic profile 与 full recognition profile
6. 明确 basic profile 下 manual_wave_driver 机制
7. 明确执行器不支持 action / condition / verify / retry 增强时的行为
8. 明确 metadata.json、custom_steps、color_rules 的定位
9. 补充地图配置器三层校验规则
10. 给出 basic profile 示例
```

# 2. V1.1.1 核心结论

V1.1.1 冻结以下补充规则：

```text
1. 导出的 script.json 中所有 ratio 坐标均基于 GameViewportRect。
2. 地图配置器内部可以使用底图像素坐标、content_rect 相对坐标或编辑器坐标。
3. content_rect 仅作为配置器内部编辑、底图标定和导出转换依据。
4. 执行器不得依赖 content_rect 解释运行时坐标。
5. V1.1 协议保留 floors/default_floor_id/floor_id。
6. basic profile 仅要求支持单楼层。
7. 多楼层脚本需要 multi_floor capability。
8. slot 相关高级动作默认 ensure_region(slot.region_id)。
9. pan_to_region 是可选显式导航动作，不是 place_trap 的必需前置动作。
10. compatibility 必须可由 action / condition / verify / retry / map 特性推导。
11. optional_capabilities 只表示增强能力，不能承载 enabled=true action 的必要执行语义。
12. basic profile 可使用 manual_wave_driver，不要求 wave_ocr。
13. basic 示例不依赖 OCR、slot visual verify、trap level verify、upgrade_trap、remove_trap。
14. full recognition 示例如果启用识别、校验、升级等能力，必须补齐 required_capabilities。
15. metadata.json 为 optional/editor-only。
16. custom_steps 为 experimental。
17. color_rules 为 optional recognition resource。
```

# 3. 坐标体系修订

## 3.1 基本原则

V1.1.1 继续保持 V1.1 的核心原则：

```text
导出的 script.json 中，所有 ratio 坐标均基于 GameViewportRect。
```

`GameViewportRect` 表示：

```text
执行器实际截图区域 / 游戏有效画面区域。
```

执行器使用统一换算公式：

```text
screen_x = viewport_left + viewport_width  × x_ratio
screen_y = viewport_top  + viewport_height × y_ratio
screen_w = viewport_width  × w_ratio
screen_h = viewport_height × h_ratio
```

---

## 3.2 配置器内部坐标与导出坐标

地图配置器内部可以使用以下任意坐标表示：

```text
1. 底图原始像素坐标
2. 编辑器画布坐标
3. floor content_rect 相对坐标
4. 当前画布 scale/pan 后的临时视图坐标
5. 其他内部编辑坐标
```

但导出的 `script.json` 中，所有以下字段均必须转换为 `GameViewportRect` ratio：

```text
1. recognition.rois
2. region.enter_actions 中 drag_ratio 的 from / to
3. slot.position
4. slot.verify.check_area
5. action 中直接出现的 ratio 点或 ratio 矩形
6. 后续扩展字段中的 x_ratio / y_ratio / w_ratio / h_ratio
```

---

## 3.3 content_rect 的定位

如果地图配置器使用 `map.calibration.content_rect` 或类似字段进行底图有效区域标定，则：

```text
1. content_rect 仅用于配置器内部编辑、底图标定和导出转换。
2. content_rect 不改变 script.json 中 ratio 坐标基于 GameViewportRect 的语义。
3. 执行器不得根据 content_rect 重新解释 slot.position。
4. 执行器不得依赖配置器编辑视口的 scale / pan。
5. content_rect 如需写入脚本包，只能作为 editor/debug 元数据。
```

推荐：

```text
project.json 可以保存 content_rect。
script.json 默认不应依赖 content_rect。
```

---

## 3.4 content_rect 到 GameViewportRect ratio 的导出转换

如果配置器内部 slot 坐标是相对 `content_rect` 的比例坐标，则导出时应转换为 `GameViewportRect` ratio。

假设底图尺寸为：

```text
image_width
image_height
```

有效区域为：

```json
{
  "x": 100,
  "y": 80,
  "w": 1720,
  "h": 900
}
```

内部槽位坐标为：

```json
{
  "x_ratio": 0.5,
  "y_ratio": 0.5
}
```

导出坐标为：

```text
export_x_ratio = (content_rect.x + content_rect.w × internal_x_ratio) / image_width
export_y_ratio = (content_rect.y + content_rect.h × internal_y_ratio) / image_height
```

矩形区域转换为：

```text
export_x_ratio = (content_rect.x + content_rect.w × internal_x_ratio) / image_width
export_y_ratio = (content_rect.y + content_rect.h × internal_y_ratio) / image_height
export_w_ratio = (content_rect.w × internal_w_ratio) / image_width
export_h_ratio = (content_rect.h × internal_h_ratio) / image_height
```

---

## 3.5 推荐实现原则

```text
1. project.json 可以保存配置器内部坐标。
2. script.json 只能保存执行器协议坐标。
3. Exporter 负责完成内部坐标到 GameViewportRect ratio 的转换。
4. Importer 如需重新编辑 script.json，可将 GameViewportRect ratio 反算回编辑器内部坐标。
5. 高级 JSON 编辑器如果编辑的是导出协议对象，应提示用户该 JSON 坐标已经是 GameViewportRect ratio。
```

# 4. 多楼层协议修订

## 4.1 基本结论

V1.1.1 在协议层保留多楼层字段。

但执行器 basic profile 只要求支持单楼层。

多楼层属于扩展能力，需要 capability：

```text
multi_floor
```

---

## 4.2 map 结构修订

`map` 推荐结构：

```json
{
  "map_id": "space_station",
  "map_name": "空间站",
  "difficulty": "normal",
  "strategy_id": "baseline",
  "base_resolution": {
    "width": 1920,
    "height": 1080
```

| `default_floor_id` | string | 否 | 默认楼层 ID |
| `floors[].floor_id` | string | 是 | 楼层唯一 ID |
| `floors[].name` | string | 否 | 楼层展示名称 |
| `floors[].description` | string | 否 | 楼层说明 |

## 4.6 多楼层兼容规则

```text
1. 如果 map.floors 缺失，则视为单楼层 legacy 脚本。
2. 如果 map.floors 只有一个楼层，basic profile 可以支持。
3. 如果 map.floors 多于一个楼层，则脚本 required_capabilities 必须包含 multi_floor。
4. 如果执行器不支持 multi_floor，应拒绝执行多楼层脚本。
5. slot.floor_id 缺失时，默认继承 map.default_floor_id。
6. region.floor_id 缺失时，默认继承 map.default_floor_id。
7. slot.region_id 引用的 region 应与 slot.floor_id 一致。
```

# 5. region 进入语义修订

## 5.1 基本结论

V1.1.1 采用以下语义：

```text
slot 相关高级动作默认自动 ensure_region(slot.region_id)。
```

包括：

```text
1. place_trap
2. remove_trap
3. click_slot，若后续定义
4. slot_has_trap verify，若需要实际进入 region
5. slot_empty verify，若需要实际进入 region
```

`pan_to_region` 保留，但作为显式导航动作，不是 `place_trap` 的必需前置动作。

---

## 5.2 ensure_region 语义

对于依赖 slot 的动作，执行器应在执行动作前确保当前视野已经进入目标 slot 所属 region。

例如执行：

```json
{
  "type": "place_trap",
  "slot_id": "A01"
}
```

执行器应解析：

```text
A01.slot_id
  ↓
slot.region_id = entrance_left
  ↓
region.enter_policy / region.enter_actions
  ↓
进入 entrance_left 对应视野
```

然后再执行点击。

---

## 5.3 current_region_id 缓存

执行器可以维护：

```text
current_region_id
```

如果当前已处于目标 region，则可跳过重复执行 `region.enter_actions`。

规则：

```text
1. reset_view_to_origin 后，current_region_id 应更新为 map.initial_view.origin_region_id。
2. 成功执行 pan_to_region 后，current_region_id 应更新为目标 region_id。
3. 成功执行 place_trap 的 ensure_region 后，current_region_id 应更新为 slot.region_id。
4. 如果执行器无法确认当前 region，应将 current_region_id 置为空，并重新执行 ensure_region。
5. 如果动作失败或发生人工干预，执行器可以清空 current_region_id。
```

---

## 5.4 pan_to_region 的定位

`pan_to_region` 是显式导航动作，适用于：

```text
1. 调试 region.enter_actions
2. 在日志中明确记录区域切换
3. 多个后续动作共享同一 region
4. 在复杂脚本中主动校正当前视野
```

普通 `place_trap` 动作不必显式添加 `pan_to_region`。

---

## 5.5 place_trap 默认语义修订

`place_trap` 默认执行语义修订为：

```text
1. 确保地图界面打开。
2. 根据 slot_id 找到 slot。
3. 执行 ensure_region(slot.region_id)。
4. 检查 conditions。
5. 按 trap.select_key。
6. 根据 slot.position 计算 GameViewportRect 坐标。
7. 点击该位置。
8. 等待 runtime.wait_after_place_ms。
9. 如果 verify.required=true，则执行校验。
10. 失败按 retry 和 on_fail 处理。
```

# 6. Compatibility Capability 推导规则

## 6.1 基本原则

执行器和地图配置器应根据脚本实际使用的 action、condition、verify、retry 特性推导所需 capability。

推导规则用于：

```text
1. 地图配置器导出前 compatibility 校验
2. 执行器加载脚本时 compatibility 校验
3. 判断脚本是否可在当前执行器 profile 下运行
4. 生成 warning / error
```

---

## 6.2 通用规则

```text
1. enabled=false 的 action 不参与 required capability 推导。
2. enabled=true 的 action 使用的必要能力必须包含在 required_capabilities。
3. optional_capabilities 只表示可增强能力，不得承载 action 成功所必需的能力。
4. verify.required=false 可依赖 optional_capabilities。
5. verify.required=true 必须依赖 required_capabilities。
6. 如果执行器不支持 enabled=true action 的 type，应拒绝执行。
7. 如果执行器不支持 enabled=true action 中 required=true 的 verify，应拒绝执行。
8. 如果执行器不支持 enabled=true action 中的必要 condition，应拒绝执行或由地图配置器阻止导出。
```

---

## 6.3 Action capability 映射

| 使用内容 | 所需 capability |
|---|---|
| `action.type=log` | `log_action` |
| `action.type=wait` | `wait_action` |
| `action.type=pan_to_region` | `pan_to_region` |
| `action.type=place_trap` | `place_trap` |
| `action.type=upgrade_trap` | `upgrade_trap` |
| `action.type=remove_trap` | `remove_trap` |
| `execute.method=custom_steps` | `custom_steps` |

---

## 6.4 Region enter action capability 映射

| 使用内容 | 所需 capability |
|---|---|
| `reset_view_to_origin` | `reset_view_to_origin` |
| `drag_ratio` | `pan_to_region` |
| `wait` | `wait_action` |

说明：

```text
drag_ratio 作为 region enter action 时，视为 pan_to_region 能力的一部分。
```

---

## 6.5 Condition capability 映射

| 条件类型 | 所需 capability |
|---|---|
| `wave_eq` | `condition_wave` |
| `wave_gte` | `condition_wave` |
| `resource_gte` | `condition_resource` + `resource_ocr` |
| `slot_empty` | `condition_slot_state` + `slot_visual_verify` |
| `slot_occupied` | `condition_slot_state` + `slot_visual_verify` |
| `trap_level_lt` | `condition_trap_level` + `trap_level_verify` |
| `trap_level_gte` | `condition_trap_level` + `trap_level_verify` |

说明：

```text
condition_basic 只表示执行器支持 conditions.operator/items 的基础结构和 operator=all。
condition_basic 不代表执行器具备 resource、wave、slot_state、trap_level 等具体状态识别能力。
```

---

## 6.6 Verify capability 映射

| Verify 类型 | required=true 时所需 capability |
|---|---|
| `slot_has_trap` | `slot_visual_verify` + `template_matching` |
| `slot_empty` | `slot_visual_verify` |
| `trap_level_gte` | `trap_level_verify` |

规则：

```text
1. verify.required=true 时，如果执行器不支持对应 capability，应 compatibility 校验失败。
2. verify.required=false 时，如果执行器不支持对应 capability，可记录 warning 并跳过 verify。
```

---

## 6.7 Retry capability 映射

| retry 字段 | 所需 capability |
|---|---|
| `max_attempts` / `interval_ms` | `retry_basic` |
| `reset_view_before_retry=true` | `retry_reset_view` + `reset_view_to_origin` |
| `micro_adjust_on_retry=true` | `retry_micro_adjust` + `micro_adjust` |

规则：

```text
1. retry.max_attempts 表示总尝试次数，包含第一次执行。
2. retry_basic 只覆盖 max_attempts / interval_ms。
3. reset_view_before_retry 和 micro_adjust_on_retry 是增强 retry 能力，需要额外 capability。
```

---

## 6.8 Recognition capability 映射

| 使用内容 | 所需 capability |
|---|---|
| `recognition.templates` 被 required verify 使用 | `template_matching` |
| `recognition.color_rules` 被 required verify 使用 | `slot_visual_verify` |
| `recognition.multi_frame` 被实际使用 | `multi_frame_vote` |
| `recognition.rois.wave` 被自动波次识别使用 | `wave_ocr` |
| `recognition.rois.resource` 被资源识别使用 | `resource_ocr` |
| `recognition.rois.core_hp` 被核心血量识别使用 | `core_hp_ocr` |

---

## 6.9 多楼层 capability 映射

| 使用内容 | 所需 capability |
|---|---|
| `map.floors.length > 1` | `multi_floor` |

# 7. Basic Profile 修订

## 7.1 基本定位

`tower_defense_v1_basic` 用于第一阶段执行器落地。

目标是支持：

```text
1. 脚本包加载
2. script.json 解析
3. schema_version / game_mode 校验
4. compatibility 基础校验
5. GameViewportRect ratio 坐标换算
6. enabled=false 跳过
7. action_id 日志
8. log action
9. wait action
10. pan_to_region
11. place_trap
12. retry_basic
13. dry-run 动作执行
14. manual_wave_driver
```

---

## 7.2 basic profile capability 建议

```json
{
  "executor_min_version": "0.2.0",
  "profile": "tower_defense_v1_basic",
  "required_capabilities": [
    "script_load",
    "script_validate",
    "viewport_ratio_coordinate",
    "manual_wave_driver",
    "map_open_close",
    "reset_view_to_origin",
    "pan_to_region",
    "place_trap",
    "wait_action",
    "log_action",
    "retry_basic",
    "condition_basic",
    "dry_run"
  ],
  "optional_capabilities": [
    "window_capture",
    "map_ui_detection",
    "template_matching",
    "resource_ocr",
    "wave_ocr",
    "slot_visual_verify",
    "trap_level_verify",
    "micro_adjust",
    "upgrade_trap",
    "remove_trap"
  ]
}
```

---

## 7.3 新增 capability：manual_wave_driver

```text
manual_wave_driver
```

含义：

```text
执行器不通过 OCR 自动识别当前波次，而是由用户、CLI、测试入口或 dry-run 流程手动指定当前 wave。
```

支持方式包括：

```text
1. CLI 参数指定 wave
2. 执行器 UI 手动选择 wave
3. dry-run 按 waves 数组顺序执行
4. 测试代码直接调用 execute_wave(wave_number)
```

---

## 7.4 basic profile 下 wave.trigger 语义

在 basic profile 中，如果执行器不支持 `wave_ocr`，则 `wave.trigger` 不由 OCR 自动触发。

执行器可以通过 `manual_wave_driver` 模拟 trigger 满足。

例如：

```text
1. dry-run 模式下按 waves 数组顺序执行；
2. real-run 测试模式下由用户手动选择当前 wave；
3. 自动化测试中直接调用 execute_wave(1)。
```

脚本仍可使用：

```json
{
  "type": "wave_eq",
  "value": 1,
  "stable_frames": 3
}
```

`manual_wave_driver` 可以手动模拟该 `wave_eq` 条件满足。

---

## 7.5 basic profile 不强制支持内容

basic profile 不强制支持：

```text
1. wave_ocr
2. resource_ocr
3. slot_visual_verify
4. trap_level_verify
5. template_matching 的真实执行
6. upgrade_trap
7. remove_trap
8. micro_adjust
9. multi_floor
10. custom_steps
```

# 9. 不支持能力时的执行器行为

## 9.1 加载前校验

执行器加载脚本时，应先进行：

```text
1. schema_version 校验
2. game_mode 校验
3. package 校验
4. semantic 校验
5. compatibility 校验
```

---

## 9.2 不支持 enabled=true action

如果脚本中存在：

```json
{
  "enabled": true,
  "type": "remove_trap"
}
```

但执行器不支持对应 capability：

```text
执行器应拒绝执行脚本。
```

如果 action 为：

```json
{
  "enabled": false,
  "type": "remove_trap"
}
```

则执行器可以加载脚本，执行时跳过该 action，并记录日志。

---

## 9.3 不支持 condition

如果 `enabled=true` 的 action 中包含执行器不支持的 condition：

```text
real-run：应视为 compatibility 校验失败，拒绝执行。
dry-run：可以记录 condition=unknown，并根据 dry-run 策略继续模拟。
```

推荐默认策略：

```text
正式执行前拒绝。
```

---

## 9.4 不支持 verify

根据 `verify.required` 处理：

```text
required=true：
  如果执行器不支持对应 verify capability，应 compatibility 校验失败。

required=false：
  如果执行器不支持对应 verify capability，可以记录 warning，并跳过 verify。
```

---

## 9.5 不支持 retry 增强

例如：

```json
"micro_adjust_on_retry": true
```

但执行器不支持 `micro_adjust`。

处理建议：

```text
1. 如果执行器配置为 strict compatibility，则拒绝执行。
2. 如果执行器配置为 permissive compatibility，则记录 warning，并降级为不执行 micro_adjust。
```

地图配置器导出前也应提供类似选项：

```text
1. error：阻止导出
2. warning：允许导出
```

# 13. 地图配置器校验规则补充

地图配置器继续实现三层校验：

```text
1. Schema Validator
2. Semantic Validator
3. Executor Compatibility Validator
```

---

## 13.1 Schema Validator 补充规则

```text
1. map.floors 如果存在，必须是数组。
2. map.default_floor_id 如果存在，必须是 string。
3. slot.floor_id 如果存在，必须是 string。
4. region.floor_id 如果存在，必须是 string。
5. trigger.type 必须是已知枚举。
6. action.enabled 如果缺失，默认 true。
7. retry.micro_adjust_on_retry 如果存在，必须是 boolean。
8. retry.reset_view_before_retry 如果存在，必须是 boolean。
9. x_ratio / y_ratio / w_ratio / h_ratio 必须在 0 到 1 之间。
10. duration_ms 必须 >= 0。
11. max_attempts 必须 >= 1。
```

---

## 13.2 Semantic Validator 补充规则

```text
1. trap_id 唯一。
2. region_id 唯一。
3. slot_id 唯一。
4. template_id 唯一。
5. action_id 在全脚本内唯一。
6. enabled=false 的 action 允许缺少部分执行字段，但必须有 type/action_id/name。
7. enabled=true 的 action 必须满足对应 action type 的必填字段。
8. map.default_floor_id 如果存在，必须引用 map.floors 中存在的 floor_id。
9. slot.floor_id 如果存在，必须引用 map.floors 中存在的 floor_id。
10. region.floor_id 如果存在，必须引用 map.floors 中存在的 floor_id。
11. slot.region_id 引用的 region 必须存在。
12. 如果 slot.floor_id 和 region.floor_id 均存在，则二者必须一致。
13. slot.default_trap 如果存在，必须引用 traps 中存在的 trap_id。
14. place_trap.trap_id 必须存在。
15. place_trap.slot_id 必须存在。
16. upgrade_trap.trap_id 必须存在。
17. remove_trap.slot_id 必须存在。
18. verify.slot_id 如果存在，必须引用 slots 中存在的 slot_id。
19. verify.trap_id 如果存在，必须引用 traps 中存在的 trap_id。
20. conditions 中引用的 slot_id 必须存在。
21. conditions 中引用的 trap_id 必须存在。
22. trap.recognition.placed_templates 引用的 template_id 必须存在。
23. trap.recognition.icon_template 如果非空，必须引用 template_id。
24. trap.recognition.level_templates 中的 template_id 必须存在。
25. slot.verify.empty_color_rule 如果非空，必须引用 color_rules.rule_id。
26. recognition.templates.path 不能为空。
27. region.enter_actions 不得包含未知 action。
28. enabled=true 的 remove_trap 如果 execute.method=custom_steps，则 steps 不得为空。
29. upgrade_trap.target_level 不得超过 trap.max_level。
30. map.floors 多于一个楼层时，required_capabilities 必须包含 multi_floor。
```

---

## 13.3 Executor Compatibility Validator 补充规则

```text
1. 根据 enabled=true action 推导 required capability。
2. 根据 conditions 推导 required capability。
3. 根据 verify.required=true 推导 required capability。
4. 根据 retry 增强字段推导 required capability。
5. 根据 map.floors 使用情况推导 multi_floor。
6. 如果推导能力不在 required_capabilities 中，则提示配置错误。
7. 如果目标执行器 profile 不支持 required_capabilities，则阻止导出或产生 warning。
8. optional_capabilities 不得承载 enabled=true action 的必要执行能力。
9. enabled=false action 不参与 required capability 推导。
10. verify.required=false 可产生 optional capability warning，但不阻止 basic 导出。
```

# 15. 对原 V1.1 示例的修订建议

```text

8. 如果 map.floors 多于一个楼层，则 required_capabilities 必须包含 multi_floor。
```

# 17. V1.1.1 最终结论

