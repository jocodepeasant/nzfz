# 1. 文档说明

本文档用于描述《逆战：未来》塔防模式自动化通关测试系统的 V1.1 需求、架构设计、模块划分、技术栈建议，以及地图配置器与自动化执行器之间的脚本 JSON 协议。

系统拆分为两个核心子系统：

```text
1. 地图配置器
2. 自动化执行器
```

两者之间只通过脚本包通信。

```text
地图配置器负责定义和导出脚本。
自动化执行器只负责加载、校验和执行脚本。
```

# 4. 总体架构设计

## 4.1 总体架构

```text
地图配置器
  ├── 脚本基础信息配置
  ├── compatibility 配置
  ├── 脚本资源包管理
  ├── 识别区域配置
  ├── 模板资源配置
  ├── 颜色规则配置
  ├── 陷阱配置
  ├── 陷阱识别资源绑定
  ├── 地图区域配置
  ├── 区域进入动作配置
  ├── 陷阱格子配置
  ├── 波次动作配置
  ├── 条件配置
  ├── 校验与失败策略配置
  ├── JSON Schema 校验
  ├── 语义校验
  ├── 执行器兼容性校验
  └── 导出脚本包
                 ↓
            脚本包
        script.json + resources
                 ↓
    自动化执行器
      ├── 脚本包加载
      ├── 脚本 JSON 校验
      ├── capability 校验
      ├── 游戏窗口管理
      ├── GameViewportRect 管理
      ├── 屏幕采集
      ├── OCR / 图像识别
      ├── 状态管理
      ├── 条件判断
      ├── 地图区域导航
      ├── 动作执行
      ├── 失败重试
      ├── 日志报告
      └── 批量跑局
```

---

## 4.2 设计原则

| 原则 | 说明 |
|---|---|
| 配置与执行解耦 | 配置器只生成脚本包，执行器只消费脚本包 |
| 脚本描述意图 | 脚本描述要做什么，不决定是否允许真实点击 |
| 安全配置本地化 | dry-run、真实输入、安全校验由执行器本地决定 |
| 地图适配配置化 | 新地图尽量通过配置器适配，不改执行代码 |
| 坐标比例化 | 坐标以比例形式保存，基于 GameViewportRect |
| 视野语义明确 | slot.position 表示进入 region 后的视口比例坐标 |
| 动作原子化 | 放置、升级、拆除、等待、拖拽抽象为动作 |
| 条件结构化 | conditions 使用 operator/items 标准结构 |
| 校验结构化 | verify 明确类型、目标和 required 策略 |
| 重试语义明确 | retry.max_attempts 表示总尝试次数 |
| 能力可声明 | 脚本声明 required_capabilities |
| 编辑器强校验 | 导出前进行 schema、semantic、compatibility 三层校验 |
| 可扩展 | BOSS、队列、多策略、OCR、NMS 等功能预留扩展点 |

```json

  "type": "directory",

  "template_dir": "templates",
  "preview_dir": "previews",
  "path_rule": "relative_to_script_json"
```

## 6.4 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `type` | string | 是 | 脚本包类型，V1.1 推荐 `directory` |
| `resource_base` | string | 是 | 资源根目录，相对 script.json |
| `template_dir` | string | 是 | 模板目录 |
| `preview_dir` | string | 否 | 预览截图目录 |
| `path_rule` | string | 是 | 路径规则，推荐 `relative_to_script_json` |

---

## 6.5 路径规则

所有资源路径默认相对于：

```text
script.json 所在目录
```

例如：

```json
{
  "template_id": "map_ui_indicator",
  "path": "templates/map_ui_indicator.png"
}
```

实际路径为：

```text
<script.json 所在目录>/templates/map_ui_indicator.png
```

# 9. 坐标体系规范

## 9.1 核心原则

V1.1 规定：

```text
所有 ratio 坐标均基于 GameViewportRect。
```

`GameViewportRect` 表示：

```text
执行器实际截图区域 / 游戏有效画面区域。
```

不应直接使用 Windows 外框窗口矩形。

---

## 9.2 GameViewportRect

```text
GameViewportRect:
  left   = 游戏截图区域左上角屏幕 x
  top    = 游戏截图区域左上角屏幕 y
  width  = 游戏截图区域宽度
  height = 游戏截图区域高度
```

点坐标换算：

```text
screen_x = viewport_left + viewport_width  × x_ratio
screen_y = viewport_top  + viewport_height × y_ratio
```

矩形坐标换算：

```text
screen_x = viewport_left + viewport_width  × x_ratio
screen_y = viewport_top  + viewport_height × y_ratio
screen_w = viewport_width  × w_ratio
screen_h = viewport_height × h_ratio
```

---

## 9.3 ViewportPointRatio

```json
{
  "x_ratio": 0.452,
  "y_ratio": 0.561
}
```

含义：

```text
基于 GameViewportRect 的比例点坐标。
```

---

## 9.4 ViewportRectRatio

```json
{
  "x_ratio": 0.435,
  "y_ratio": 0.545,
  "w_ratio": 0.035,
  "h_ratio": 0.035
}
```

含义：

```text
基于 GameViewportRect 的比例矩形区域。
```

---

## 9.5 slot.position 语义

```text
slot.position 表示进入 slot.region_id 对应视野之后，slot 在 GameViewportRect 内的比例坐标。
```

执行 `place_trap(slot=A01)` 时：

```text
1. 进入 A01.region_id 对应区域
2. 根据 slot.position 计算屏幕坐标
3. 点击该位置
```

它不是：

```text
世界地图坐标
region 内局部坐标
Windows 外框窗口坐标
```

---

## 9.6 recognition.rois 语义

```text
recognition.rois 中的 ROI 是固定 UI 区域，一般不依赖 region。
```

例如：

```text
wave
resource
core_hp
map_ui_indicator
place_error_tip
```

---

## 9.7 slot.verify.check_area 语义

```text
slot.verify.check_area 表示进入 slot.region_id 对应视野之后，用于判断该 slot 状态的 viewport ratio 矩形。
```


---

## 20.4 执行语义

### policy=skip

```text
条件不满足时，跳过当前 action，继续后续 action。
```

### policy=fail

```text

```

### policy=wait_until_met

```text
条件不满足时，在 timeout_ms 内每隔 check_interval_ms 重新检查。
如果条件满足，继续执行 action。
如果超时，根据 on_timeout 处理。
```

# 24. 地图编辑器校验规则

地图编辑器必须实现三层校验：

```text
1. Schema Validator
2. Semantic Validator
3. Executor Compatibility Validator
```

---

## 24.1 Schema Validator

校验：

```text
字段类型
必填字段
枚举值
数字范围
数组结构
对象结构
```

示例规则：

```text
1. schema_version 必须存在
2. game_mode 必须为 tower_defense
3. x_ratio/y_ratio/w_ratio/h_ratio 必须在 0 到 1 之间
4. threshold 必须在 0 到 1 之间
5. duration_ms 必须 >= 0
6. max_attempts 必须 >= 1
7. traps 必须是数组
8. waves 必须是数组
```

---

## 24.2 Semantic Validator

校验引用关系和业务逻辑：

```text
1. trap_id 唯一
2. region_id 唯一
3. slot_id 唯一
4. template_id 唯一
5. action_id 在脚本内唯一
6. slot.region_id 必须存在
7. slot.default_trap 必须存在
8. place_trap.trap_id 必须存在
9. place_trap.slot_id 必须存在
10. upgrade_trap.trap_id 必须存在
11. remove_trap.slot_id 必须存在
12. verify.slot_id 必须存在
13. verify.trap_id 必须存在
14. conditions 中引用的 slot_id 必须存在
15. conditions 中引用的 trap_id 必须存在
16. trap.recognition 引用的 template_id 必须存在
17. recognition.templates.path 不能为空
18. region.enter_actions 不得包含未知 action
19. remove_trap custom_steps 不能为空
20. upgrade_trap.target_level 不得超过 trap.max_level
21. resource_gte 建议不低于 trap.cost 或 upgrade_cost
22. action.enabled=false 时允许缺少部分执行字段，但必须有 type/action_id/name
```

---

## 24.3 Executor Compatibility Validator

校验脚本是否使用了当前执行器不支持的能力。

例如当前执行器 profile：

```text
tower_defense_v1_basic
```

支持：

```text
log_action
wait_action
pan_to_region
place_trap
retry_basic
```

不支持：

```text
remove_trap
trap_level_verify
micro_adjust
resource_ocr
```

如果脚本使用了不支持能力，地图编辑器应提示：

```text
脚本使用了当前执行器不支持的能力：remove_trap
```

并根据配置：

```text
阻止导出
或允许以 warning 导出
```

# 25. 自动化执行器需求设计 V1.1

## 25.1 模块定位

自动化执行器负责读取地图配置器导出的脚本包，在游戏对局内执行对应自动化动作。

执行器不负责编辑任务。

执行器负责：

```text
1. 加载脚本包
2. 校验脚本
3. 校验 compatibility
```

## 25.2 执行器主流程

```text
加载脚本包
  ↓
读取 script.json
  ↓
校验 schema_version
  ↓
校验 compatibility
  ↓
校验脚本引用和资源
  ↓
等待进入对局
  ↓
检测是否在陷阱地图界面
  ↓
如果不在，按 O 打开
  ↓
记录或确认 origin
  ↓
循环识别波次 / 资源 / 血量 / 结算状态
  ↓
匹配当前 wave trigger
  ↓
检查 execute_once
  ↓
按顺序执行 wave.actions
  ↓
每个 action 执行前检查 enabled
  ↓
检查 conditions
  ↓
条件不满足则按 on_condition_failed 处理
  ↓
动作失败则按 retry / on_fail 处理
  ↓
等待下一波或下一触发条件
  ↓
识别胜利 / 失败 / 异常
  ↓
处理结算
  ↓
生成单局报告
```


# 29. V1.1 结论

V1.1 版本冻结以下核心设计：

```text
1. 地图配置器与执行器分离
2. 两者通过脚本包通信
3. 脚本包包含 script.json 和资源文件
4. schema_version 使用 1.1.0
5. 新增 compatibility 执行器能力声明
6. 新增 package 资源包定义
7. 所有 ratio 坐标基于 GameViewportRect
8. coordinate_mode 使用 viewport_ratio_after_region_enter
9. slot.position 表示进入 region 后的视口比例坐标
10. recognition 增加 rois/templates/color_rules/multi_frame
11. traps 增加 recognition 引用
12. region.enter_actions 使用 reset_view_to_origin / drag_ratio
13. conditions 使用 operator/items 标准结构
14. on_condition_failed 使用 wait_until_met/on_timeout
15. retry 使用 max_attempts
16. 所有 wave action 支持 action_id/enabled
17. 支持 wait/log 标准动作
18. 地图编辑器必须实现三层校验
19. 执行器必须检查 required_capabilities
20. 执行器第一阶段只实现最小支持子集
```

最终，V1.1 的目标不是一次性实现所有自动化能力，而是把：

```text
地图配置器输出什么
执行器能够理解什么
脚本资源如何组织
坐标如何解释
动作如何标准化
条件与重试如何表达
执行能力如何声明
```

全部规范清楚。

这样后续可以稳定推进：

```text
P2-10 塔防脚本 JSON 加载、校验与基础动作执行
P2-11 条件引擎与运行状态识别
P2-12 地图区域导航与槽位动作增强
P2-13 放置/升级/拆除陷阱完整动作
P2-14 失败重试与视觉校验
P2-15 单局报告与批量跑局
```
