## 1. 文档目标

本文档用于定义 `WindowManager` 在完成真实窗口连接能力后的断开连接与连接状态查询能力。

P0-01 到 P0-06 已经完成以下链路：

```text
数据模型
    ↓
WindowManager 基础类
    ↓
真实窗口搜索
    ↓
连接前校验
    ↓
构建 ConnectedWindow
    ↓
激活目标窗口
    ↓
保存 _connected_window
    ↓
返回 ConnectResult.success
```

P0-07 的目标是补齐连接生命周期中的最后一个 Core 层基础能力：

```text
断开当前连接，并提供稳定的状态查询接口。
```

本阶段完成后，Core 层应形成完整闭环：

```text
搜索窗口
    ↓
连接窗口
    ↓
查询连接状态
    ↓
断开连接
    ↓
回到未连接状态
```

---

## 2. 设计结论

根据当前需求边界，P0-07 最终采用以下方案：

| 项目 | 最终选择 |
|---|---|
| 实现位置 | `nzfz_executor/core/window_manager.py` |
| 对外断开接口 | `disconnect_window()` |
| `disconnect_window()` 返回值 | `None` |
| 断开是否可能失败 | 不视为失败 |
| 断开时是否调用 Windows API | 不调用 |
| 断开时是否关闭游戏窗口 | 不关闭 |
| 断开时是否杀进程 | 不杀进程 |
| 断开时是否清空 `_connected_window` | 是 |
| 断开时是否清空 `_last_error` | 是 |
| 断开时是否清空 `_last_window_info` | 否 |
| 断开时是否清空搜索结果 | 否，Core 层不保存搜索结果列表 |
| 是否新增 `DisconnectResult` | 否 |
| 是否新增连接状态查询接口 | 是 |
| 查询接口形式 | 只读属性与便捷方法 |
| 是否允许 UI 直接访问私有字段 | 不允许 |

## 3. 模块边界

### 3.1 P0-07 负责内容

P0-07 负责以下内容：

- 提供断开连接接口；
- 清理当前连接窗口状态；
- 清理最近错误信息；
- 保留最近一次连接使用的搜索快照；
- 提供当前连接状态查询能力；
- 提供当前连接窗口对象查询能力；
- 提供最近错误信息查询能力；
- 保证外部模块不需要访问 `WindowManager` 私有字段。

---

### 3.2 P0-07 不负责内容

P0-07 不实现以下能力：

- 不关闭游戏窗口；
- 不最小化游戏窗口；
- 不恢复游戏窗口；
- 不杀死游戏进程；
- 不断开系统级窗口关系；
- 不释放 Windows 句柄；
- 不调用 `DestroyWindow`；
- 不调用 `CloseWindow`；
- 不发送 `WM_CLOSE`；
- 不调用 `PostMessage` 或 `SendMessage` 关闭窗口；
- 不清理 GUI 搜索结果；
- 不控制 GUI 按钮状态；
- 不展示提示弹窗；
- 不执行健康检测；
- 不执行自动重连；
- 不停止自动化任务；
- 不处理多窗口会话；
- 不负责执行器主流程调度。

## 4. 断开连接语义

### 4.1 方法定义

`WindowManager` 需要提供以下公开方法：

```python
def disconnect_window(self) -> None:
    ...
```

---

### 4.2 方法职责

`disconnect_window()` 的职责是：

```text
清除 WindowManager 内部保存的当前连接窗口对象，使其回到未连接状态。
```

具体来说，需要清理：

```python
self._connected_window
```

断开后：

```python
self._connected_window is None
```

必须成立。

---

### 4.3 断开连接不等于关闭窗口

断开连接只是解除执行器内部对目标窗口的绑定关系。

它不代表：

- 关闭游戏窗口；
- 结束游戏进程；
- 隐藏游戏窗口；
- 修改窗口前后台状态；
- 修改窗口大小；
- 修改窗口位置；
- 释放操作系统窗口资源。

断开连接后，目标游戏窗口应继续正常存在和运行。

---

### 4.4 断开连接是否检查窗口仍然有效

不检查。

原因：

1. 断开连接是内部状态清理操作；
2. 即使目标窗口已经关闭，断开也应该可以正常完成；
3. 断开不依赖 Windows API；
4. 断开不应该因为窗口状态异常而失败；
5. 用户或上层模块需要一种稳定方式回到未连接状态。

因此，无论当前连接窗口是否仍然有效，调用：

```python
disconnect_window()
```

都应使 `WindowManager` 进入未连接状态。

---

### 4.5 断开连接是否有失败结果

没有。

`disconnect_window()` 不返回 `ConnectResult`，也不新增 `DisconnectResult`。

原因：

1. 断开连接不依赖外部系统资源；
2. 断开连接本质是内存状态清理；
3. P0 阶段没有必要引入额外结果模型；
4. GUI 层可以在调用成功后自行显示“已断开连接”；
5. 简化 Core 层接口。

最终接口：

```python
def disconnect_window(self) -> None:
    self._connected_window = None
    self._last_error = None
```

---

## 5. 内部状态处理规则

### 5.1 `_connected_window`

字段语义：

```text
当前已连接窗口上下文。
```

断开时必须清空：

```python
self._connected_window = None
```

断开后：

```python
self.is_connected() is False
```

---

### 5.2 `_last_error`

字段语义：

```text
最近一次 Core 操作产生的错误信息。
```

断开连接不是错误，因此断开时应清空：

```python
self._last_error = None
```

不建议设置为：

```text
已断开连接
```

原因：

1. `_last_error` 应保持“错误信息”语义；
2. “已断开连接”不是错误；
3. 普通提示应由 GUI 层或调用方自行处理；
4. 避免 UI 把普通状态误判为异常。

---

### 5.3 `_last_window_info`

字段语义：

```text
最近一次连接使用的搜索结果快照。
```

断开连接时不清空：

```python
self._last_window_info 保持不变
```

原因：

1. 它是历史快照；
2. 它不代表当前连接状态；
3. 保留有利于调试；
4. 保留有利于后续健康检测或日志分析；
5. 是否已连接只能由 `_connected_window` 判断。

必须明确：

```text
_last_window_info 不代表当前已连接窗口。
_connected_window 才代表当前连接状态。
```

---

### 5.4 搜索结果列表

当前 Core 层 `WindowManager` 不负责保存完整搜索结果列表。

因此断开连接不会清理 GUI 中的搜索结果。

如果 GUI 层保存了搜索结果表格，是否清空由 GUI 层自行决定。

推荐 GUI 行为是：

```text
断开连接后保留搜索结果。
```

但该行为不属于 P0-07 Core 层职责。

---

## 6. 连接状态查询接口

### 6.1 设计目标

外部模块不应直接访问以下私有字段：

```python
self._connected_window
self._last_error
self._last_window_info
```

因此 `WindowManager` 需要提供稳定的公开查询接口。

---

### 6.2 当前是否已连接

推荐提供：

```python
def is_connected(self) -> bool:
    ...
```

语义：

```text
判断当前是否存在已连接窗口。
```

实现规则：

```python
def is_connected(self) -> bool:
    return self._connected_window is not None
```

注意：

```text
is_connected() 只判断当前是否保存了连接对象。
它不执行健康检测。
它不重新验证 hwnd 是否仍然有效。
它不检查进程是否仍然存活。
```

如果需要判断窗口当前是否仍然健康，应由后续健康检测需求负责。

---

### 6.3 获取当前连接窗口

推荐提供只读属性：

```python
@property
def connected_window(self) -> ConnectedWindow | None:
    ...
```

语义：

```text
返回当前已连接窗口上下文。
```

实现规则：

```python
@property
def connected_window(self) -> ConnectedWindow | None:
    return self._connected_window
```

注意：

当前 `ConnectedWindow` 按既有设计是普通 `dataclass`，允许后续刷新。

因此 P0-07 阶段可以直接返回对象引用。

后续如果需要更强封装，可以再考虑：

- 返回浅拷贝；
- 返回只读快照；
- 将可变刷新逻辑集中到 `WindowManager` 内部。

但 P0 阶段不做过度设计。

---

### 6.4 获取最近错误

推荐提供只读属性：

```python
@property
def last_error(self) -> str | None:
    ...
```

语义：

```text
返回最近一次 Core 操作产生的错误信息。
```

实现规则：

```python
@property
def last_error(self) -> str | None:
    return self._last_error
```

断开连接后：

```python
self.last_error is None
```

---

### 6.5 获取最近一次连接使用的搜索快照

可选提供只读属性：

```python
@property
def last_window_info(self) -> WindowInfo | None:
    ...
```

语义：

```text
返回最近一次成功进入连接流程时使用的 WindowInfo 快照。
```

推荐实现：

```python
@property
def last_window_info(self) -> WindowInfo | None:
    return self._last_window_info
```

注意：

```text
last_window_info 只是历史搜索快照。
它不代表当前连接仍然存在。
它也不代表窗口当前仍然有效。
```

如果调用方要判断是否已连接，必须使用：

```python
is_connected()
```

或：

```python
connected_window is not None
```

## 8. 推荐实现

以下为推荐实现片段。

```python
from __future__ import annotations

from .models import ConnectedWindow, WindowInfo


class WindowManager:
    def __init__(self) -> None:
        self._connected_window: ConnectedWindow | None = None
        self._last_window_info: WindowInfo | None = None
        self._last_error: str | None = None

    def disconnect_window(self) -> None:
        """
        断开当前窗口连接。

        该方法只清理执行器内部连接状态，不关闭窗口、不杀进程、
        不调用 Windows API，也不清理最近一次连接使用的搜索快照。
        """
        self._connected_window = None
        self._last_error = None

    def is_connected(self) -> bool:
        """
        判断当前是否存在已连接窗口。

        注意：
        该方法只判断内部是否保存 ConnectedWindow，
        不执行窗口有效性检测。
        """
        return self._connected_window is not None

    @property
    def connected_window(self) -> ConnectedWindow | None:
        """
        当前已连接窗口上下文。

        如果当前未连接，则返回 None。
        """
        return self._connected_window

    @property
    def last_error(self) -> str | None:
        """
        最近一次 Core 操作产生的错误信息。
        """
        return self._last_error

    @property
    def last_window_info(self) -> WindowInfo | None:
        """
        最近一次连接使用的 WindowInfo 搜索快照。

        注意：
        该值不代表当前仍然已连接，也不代表窗口当前仍然有效。
        """
        return self._last_window_info
```

## 9. 与 `connect_window()` 的关系

### 9.1 连接成功后状态

当 `connect_window()` 完整通过：

```text
P0-04 连接前校验
    ↓
P0-05 构建 ConnectedWindow
    ↓
P0-06 激活目标窗口
```

并且最终成功保存：

```python
self._connected_window = connected_window
```

后，应满足：

```python
self.is_connected() is True
self.connected_window is not None
```

---

### 9.2 连接失败后状态

如果 `connect_window()` 失败：

```text
不得错误保存新的 _connected_window。
```

如果此前没有连接，则：

```python
self._connected_window is None
```

如果此前已有连接，则是否保留旧连接，取决于 P0-04 到 P0-06 已确认的连接失败处理策略。

推荐策略是：

```text
连接失败不得覆盖旧连接。
```

即：

```text
新连接成功前，不修改旧的 _connected_window。
```

---

### 9.3 已连接状态下是否允许再次连接

P0-07 建议明确以下产品级规则：

```text
P0 阶段不支持直接切换连接窗口。
如果当前已连接窗口，用户需要先断开，再连接其他窗口。
```

因此 `connect_window()` 可增加防御逻辑：

```python
if self._connected_window is not None:
    self._last_error = "当前已连接窗口，请先断开后再连接其他窗口"
    return ConnectResult.fail(self._last_error)
```

该逻辑可以放在连接流程早期，建议位于：

```text
参数校验之后，真实 Windows 校验之前。
```

这样可以避免在已连接状态下执行不必要的 Windows API 调用。

如果当前项目已有明确设计允许直接切换窗口，则本条可不采用。

但从执行器稳定性角度，P0 阶段推荐：

```text
必须先断开，再连接新窗口。
```

## 10. 状态转换设计

### 10.1 初始状态

`WindowManager` 初始化后：

```python
self._connected_window is None
self._last_window_info is None
self._last_error is None
```

对外表现：

```python
is_connected() == False
connected_window is None
last_error is None
last_window_info is None
```

---

### 10.2 连接成功状态

成功连接窗口后：

```python
self._connected_window is not None
self._last_window_info is not None
self._last_error is None
```

对外表现：

```python
is_connected() == True
connected_window is not None
last_error is None
```

---

### 10.3 连接失败状态

连接失败后：

```python
self._last_error = result.message
```

如果之前未连接：

```python
self._connected_window is None
```

如果之前已连接，并且采用“不允许直接切换”策略，则：

```python
self._connected_window 保持原值
```

对外表现：

```python
last_error is not None
```

---

### 10.4 断开后状态

调用：

```python
disconnect_window()
```

后：

```python
self._connected_window is None
self._last_error is None
```

`_last_window_info` 保持不变。

对外表现：

```python
is_connected() == False
connected_window is None
last_error is None
```

## 11. 状态转换表

| 当前状态 | 操作 | 结果状态 | 说明 |
|---|---|---|---|
| 初始未连接 | `disconnect_window()` | 未连接 | 幂等操作 |
| 初始未连接 | `connect_window()` 成功 | 已连接 | 保存 `_connected_window` |
| 初始未连接 | `connect_window()` 失败 | 未连接 | 保存 `_last_error` |
| 已连接 | `disconnect_window()` | 未连接 | 清空 `_connected_window` |
| 已连接 | `connect_window()` | 已连接或失败 | 推荐拒绝，要求先断开 |
| 已连接 | 目标窗口外部关闭 | 内部仍显示已连接 | 是否失效由后续健康检测处理 |
| 窗口已失效但内部未断开 | `disconnect_window()` | 未连接 | 不检查窗口，直接清理 |

---

## 12. 幂等性要求

`disconnect_window()` 必须是幂等的。

即多次调用：

```python
manager.disconnect_window()
manager.disconnect_window()
manager.disconnect_window()
```

不应抛出异常。

多次调用后仍满足：

```python
manager.is_connected() is False
manager.connected_window is None
manager.last_error is None
```

---

## 13. 非 Windows 平台行为

`disconnect_window()` 不依赖 Windows API。

因此在非 Windows 平台上也应正常工作。

行为与 Windows 平台一致：

```python
self._connected_window = None
self._last_error = None
```

不需要返回：

```text
当前平台不支持
```

原因：

```text
断开连接只是内部状态清理，不是平台相关操作。
```

---

## 14. 缺少依赖时行为

如果缺少：

```text
pywin32
psutil
```

`disconnect_window()` 仍应正常工作。

不检查依赖，不返回错误。

原因：

```text
断开连接不需要 Windows API，也不需要进程 API。
```

## 15. 日志建议

P0-07 可选加入基础日志。

推荐日志点：

```text
开始断开连接
当前未连接，忽略断开请求
已断开连接
```

示例：

```python
def disconnect_window(self) -> None:
    if self._connected_window is None:
        logger.debug("disconnect_window called, but no window is connected")
    else:
        logger.info(
            "disconnect window: hwnd=%s, pid=%s, title=%s",
            self._connected_window.hwnd,
            self._connected_window.pid,
            self._connected_window.title,
        )

    self._connected_window = None
    self._last_error = None
```

日志不是强制要求，具体取决于项目当前是否已经引入日志系统。

## 17. 验收标准

P0-07 完成后，应满足以下验收标准：

### 17.1 功能验收

- `WindowManager` 提供 `disconnect_window()`；
- `disconnect_window()` 可在未连接状态调用；
- `disconnect_window()` 可在已连接状态调用；
- `disconnect_window()` 调用后 `_connected_window` 被清空；
- `disconnect_window()` 调用后 `_last_error` 被清空；
- `disconnect_window()` 不清空 `_last_window_info`；
- `disconnect_window()` 不调用 Windows API；
- `disconnect_window()` 不关闭目标窗口；
- `disconnect_window()` 不杀进程；
- `disconnect_window()` 在非 Windows 平台可正常执行；
- `disconnect_window()` 在缺少 pywin32 / psutil 时可正常执行。

---

### 17.2 状态查询验收

- `is_connected()` 能正确反映是否存在 `_connected_window`；
- `connected_window` 能返回当前连接对象；
- 未连接时 `connected_window` 返回 `None`；
- `last_error` 能返回最近错误；
- 断开后 `last_error` 返回 `None`；
- `last_window_info` 能返回最近一次连接使用的搜索快照；
- 外部模块无需访问 `WindowManager` 私有字段。

---

### 17.3 幂等性验收

- 未连接时重复断开不报错；
- 已连接后断开一次进入未连接状态；
- 断开后继续重复断开仍保持未连接状态；
- 重复断开不会产生错误信息。

## 18. 风险与约束

### 18.1 风险：调用方误解 last_window_info

风险说明：

```text
调用方可能误以为 last_window_info 代表当前连接窗口。
```

规避方式：

```text
文档明确说明 last_window_info 是历史搜索快照。
是否已连接只能看 connected_window 或 is_connected()。
```

---

### 18.2 风险：断开后窗口仍存在

风险说明：

```text
用户可能误以为断开连接会关闭游戏窗口。
```

规避方式：

```text
文档和 UI 文案应明确：断开连接只是解除工具绑定，不关闭游戏。
```

---

### 18.3 风险：已连接状态下重复连接

风险说明：

```text
如果允许直接连接另一个窗口，可能导致状态覆盖和执行目标混乱。
```

推荐规避方式：

```text
P0 阶段要求先断开，再连接其他窗口。
```

Core 层可增加防御：

```python
if self._connected_window is not None:
    return ConnectResult.fail("当前已连接窗口，请先断开后再连接其他窗口")
```

## 19. 后续扩展

P0-07 不实现健康检测，但为后续 P1 健康检测提供基础状态。

后续健康检测可基于：

```python
self._connected_window
```

检查：

- `hwnd` 是否仍然有效；
- PID 是否仍然一致；
- 进程是否仍然存活；

- 窗口是否最小化；
- 窗口尺寸是否有效；
- 前台状态是否符合要求。

如果健康检测发现窗口失效，后续可选择：

- 仅返回异常状态；
- 自动清空连接；
- 提示用户断开；
- 尝试自动重连。

但这些都不属于 P0-07 范围。
