## 1. 文档目标

本文档用于定义游戏连接页的错误提示与用户反馈标准化方案。

P1-01 至 P1-04 已完成：

```text
健康检测能力
UI 异步任务
请求防覆盖、防抖与超时
游戏连接页 UI 状态机
```

但这些需求中临时引入了大量文案和提示，例如：

```text
任务执行异常：xxx
请输入窗口标题、进程名或 PID
未找到匹配窗口
搜索窗口超时，请稍后重试
连接窗口超时，请稍后重试
窗口状态检测超时
窗口已最小化，请恢复窗口后重试
当前已连接其他窗口，继续操作将断开当前连接并连接所选窗口。是否继续？
```

如果不统一处理，会导致：

- 同类提示文案不一致；
- 技术异常直接暴露给用户；
- 状态标签和错误提示混用；
- 弹框使用不克制；
- UI 文案散落在 `game_connect.py`；
- 后续维护、测试、国际化困难；
- 日志与用户提示无法对应。

P1-05 的目标是：

```text
为游戏连接页建立标准化的用户反馈体系，包括提示码、提示级别、统一文案、展示规则、样式规则和日志规则。
```

---

## 2. 设计结论

根据 P1 设计讨论，P1-05 最终采用以下方案：

| 项目 | 最终选择 |
|---|---|
| 是否新增反馈模块 | 是 |
| 新增模块 | `ui/feedback/messages.py` |
| 是否引入提示码 | 是 |
| 提示码枚举 | `FeedbackCode` |
| 是否引入提示级别 | 是 |
| 提示级别枚举 | `FeedbackLevel` |
| 反馈数据结构 | `FeedbackMessage` |
| 文案集中管理 | `FEEDBACK_MESSAGES` |
| 是否支持模板参数 | 是 |
| 获取文本方法 | `get_feedback_text()` |
| 获取级别方法 | `get_feedback_level()` |
| UI 是否继续散落硬编码文案 | 否 |
| Worker 原始异常是否直接展示 | 否 |
| Worker 原始异常处理 | 写日志 |
| Worker 用户侧提示 | 标准化文案 |
| Core 业务失败 | 优先显示 `result.message` |
| `result.message` 不存在时 | 使用标准文案 |
| 普通错误是否弹框 | 否 |
| 唯一弹框场景 | 切换连接确认 |
| 是否新增 Toast | 否 |
| 展示渠道 | 状态标签、提示标签、确认弹框、日志 |
| 样式区分 | 按 `FeedbackLevel` 设置颜色 |
| 最终国际化 | 暂不实现，但为后续预留 |

## 4. 提示展示渠道

P1-05 定义四类展示渠道。

### 4.1 状态标签

用于常驻状态展示，例如：

```text
未连接
正在连接
已连接
执行就绪
窗口未在前台
连接异常
```

特点：

- 不打断用户；
- 由 UI 状态机驱动；
- 适合连接状态和执行状态。

---

### 4.2 页面提示标签

用于短期或局部提示，例如：

```text
请输入窗口标题、进程名或 PID
未找到匹配窗口
窗口已最小化，请恢复窗口后重试
搜索窗口超时，请稍后重试
```

特点：

- 不打断用户；
- 适合搜索结果、选择提示、操作失败提示；
- 可复用搜索状态标签或 hint label。

---

### 4.3 确认弹框

仅用于需要用户明确确认的操作。

P1-05 当前唯一弹框场景：

```text
已连接状态下切换连接目标
```

普通错误不弹框。

---

### 4.4 日志

用于开发和排查问题，不直接展示给用户。

日志中可包含：

```text
原始异常
traceback
request_id
connection_generation
hwnd
pid
Worker 返回的错误字符串
过期结果丢弃原因
```

## 5. 提示级别设计

### 5.1 枚举定义

```python
class FeedbackLevel(Enum):
    INFO = "info"
    SUCCESS = "success"
    WARNING = "warning"
    ERROR = "error"
```

---

### 5.2 级别含义

| Level | 含义 | 示例 |
|---|---|---|
| `INFO` | 普通信息 | 请输入关键词 |
| `SUCCESS` | 成功状态 | 已连接，执行就绪 |
| `WARNING` | 可恢复警告 | 窗口未在前台 |
| `ERROR` | 操作失败或异常 | 连接超时 |

---

### 5.3 样式建议

| Level | 推荐颜色 |
|---|---|
| `INFO` | 灰色或蓝灰 |
| `SUCCESS` | 绿色 |
| `WARNING` | 橙色 |
| `ERROR` | 红色 |

推荐色值：

| Level | 色值 |
|---|---|
| `INFO` | `#666666` |
| `SUCCESS` | `#2e7d32` |
| `WARNING` | `#ef6c00` |
| `ERROR` | `#c62828` |

```python
def get_feedback_message(code: FeedbackCode) -> FeedbackMessage:
    return FEEDBACK_MESSAGES.get(
        code,
        FeedbackMessage(
            FeedbackLevel.ERROR,
            "发生未知错误，请稍后重试",
        ),
    )


def get_feedback_text(code: FeedbackCode, **kwargs: object) -> str:
    message = get_feedback_message(code)

    try:
        return message.text.format(**kwargs)
    except Exception:
        return message.text


def get_feedback_level(code: FeedbackCode) -> FeedbackLevel:
    return get_feedback_message(code).level

```

### 16.3 标准内容

```python
message = get_feedback_text(
    FeedbackCode.CONFIRM_SWITCH_CONNECTION_MESSAGE,
)
```

文本：

```text
当前已连接其他窗口。继续操作将断开当前连接，并连接所选窗口。是否继续？
```

---

### 16.4 推荐实现

```python
def _confirm_switch_connection(self) -> bool:
    title = get_feedback_text(
        FeedbackCode.CONFIRM_SWITCH_CONNECTION_TITLE,
    )
    message = get_feedback_text(
        FeedbackCode.CONFIRM_SWITCH_CONNECTION_MESSAGE,
    )

    reply = QMessageBox.question(
        self,
        title,
        message,
        QMessageBox.Yes | QMessageBox.No,
        QMessageBox.No,
    )

    return reply == QMessageBox.Yes
```

### 16.5 默认按钮

默认按钮必须是：

```text
否 / 取消
```

原因：

```text
切换连接会断开当前连接，属于有影响的操作。
```

## 27. 最终结论

P1-05 的核心设计结论是：

```text
游戏连接页所有用户提示应通过统一 FeedbackCode 映射到标准文案和提示级别，不再散落硬编码中文文案。
```

本阶段新增：

```text
ui/feedback/messages.py
```

并定义：

```text
FeedbackLevel
FeedbackCode
FeedbackMessage
FEEDBACK_MESSAGES
get_feedback_message()
get_feedback_text()
get_feedback_level()
```

UI 层通过：

```text
状态标签
提示标签
确认弹框
日志
```

展示反馈。

其中：

```text
普通错误不弹框；
切换连接是当前唯一弹框场景；
Worker 原始异常不直接展示给用户；
Core 业务失败优先显示 result.message；
技术细节写入日志。
```

P1-05 完成后，游戏连接页将具备统一、清晰、可维护的用户反馈体系，为后续 P2 主流程集成提供稳定 UI 基础。
