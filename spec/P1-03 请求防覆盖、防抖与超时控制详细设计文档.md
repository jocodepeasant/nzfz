## 1. 文档目标

本文档用于定义游戏连接页在引入异步 Worker 后的请求一致性控制机制。

P1-02 已完成以下能力设计：

```text
搜索窗口异步执行
连接窗口异步执行
健康检测异步执行
Worker 结果通过 Qt Signal 返回 UI
所有 Worker 信号携带 request_id
```

但异步执行引入了新的时序问题：

- 旧搜索结果可能覆盖新搜索结果；
- 连接请求迟到后可能覆盖当前状态；
- 健康检测结果可能在断开后返回并污染 UI；
- 超时后 Worker 仍可能返回结果；
- 输入框频繁变化可能产生大量搜索请求；
- 断开连接后旧连接/旧健康检测结果仍可能回来。

P1-03 的目标是：

```text
通过 request_id、connection_generation、防抖和软超时机制，确保 UI 只接受当前有效请求的结果，避免旧结果、迟到结果、断开前结果覆盖当前状态。
```

---

## 2. 设计结论

根据 P1 设计讨论，P1-03 最终采用以下方案：

| 项目 | 最终选择 |
|---|---|
| 请求防覆盖机制 | `request_id` |
| 连接生命周期防污染机制 | `connection_generation` |
| 搜索结果有效性判断 | `request_id` |
| 连接结果有效性判断 | `request_id + connection_generation` |
| 健康检测结果有效性判断 | `request_id + connection_generation` |
| `connection_generation` 递增时机 | 开始连接、断开连接 |
| 连接成功时是否递增 generation | 否 |
| 连接失败时是否递增 generation | 否 |
| 空关键词是否发起 Worker | 否 |
| 空关键词是否使旧搜索失效 | 是 |
| 输入即搜索防抖 | 300ms |
| 点击搜索按钮 | 立即搜索 |
| 输入框回车 | 立即搜索 |
| 清空输入框 | 立即清空结果并使旧搜索失效 |
| 超时策略 | 软超时 |
| 是否强杀线程 | 否 |
| 搜索超时 | 3000ms |
| 连接超时 | 5000ms |
| 健康检测超时 | 2000ms |
| 搜索超时后 | 废弃迟到结果，不影响连接 |
| 连接超时后 | 标记请求失效，并调用 `disconnect_window()` 兜底清理 |
| 迟到连接成功结果 | 丢弃，并再次调用 `disconnect_window()` 兜底 |
| 健康检测超时后 | 标记本轮结束，不改变连接生命周期 |
| 断开连接时 | 废弃旧连接和旧健康检测结果 |
| 断开连接是否废弃搜索结果 | 否 |
| 超时实现位置 | UI 层 |
| `WindowTaskRunner` 是否负责超时 | 否，保持轻量 |
| P1-03 是否实现完整 UI 状态机 | 否，P1-04 实现 |
| P1-03 是否标准化最终文案 | 否，P1-05 实现 |

## 4. 核心概念

### 4.1 `request_id`

`request_id` 是同一类异步任务的请求编号。

每类任务独立维护：

```text
搜索 request_id
连接 request_id
健康检测 request_id
```

用于判断：

```text
异步返回结果是否属于当前最新请求。
```

例如搜索：

```text
search request 1
search request 2
search request 3
```

UI 只接受：

```text
request_id == active_search_request_id
```

---

### 4.2 `connection_generation`

`connection_generation` 是连接生命周期代际编号。

用于判断异步结果是否仍属于当前连接生命周期。

例如：

```text
generation = 1，已连接窗口 A
发起健康检测 H1，记录 generation=1
用户断开连接，generation=2
H1 返回
由于 H1.generation != 当前 generation
所以丢弃 H1
```

---

### 4.3 为什么需要双机制

`request_id` 解决的是：

```text
同类任务的新旧问题。
```

`connection_generation` 解决的是：

```text
连接生命周期是否已经变化的问题。
```

两者不可互相替代。

| 场景 | 需要 request_id | 需要 generation |
|---|---:|---:|
| 连续搜索旧结果覆盖新结果 | 是 | 否 |
| 健康检测在断开后返回 | 是 | 是 |
| 连接请求在断开后返回 | 是 | 是 |
| 连接超时后迟到成功 | 是 | 是 |
| 搜索超时后迟到返回 | 是 | 否 |

---

### 4.4 软超时

P1-03 采用软超时：

```text
超时后 UI 标记请求失效并恢复状态，但不强制终止后台线程。
```

原因：

- Python/Qt 线程强杀不安全；
- Windows API 调用可能不可中断；
- 强杀可能导致资源泄露；
- 强杀可能破坏 `WindowManager` 状态。

迟到结果通过：

```text
request_id + connection_generation
```

进行废弃。

## 6. `connection_generation` 规则

### 6.1 递增时机

`connection_generation` 在以下场景递增：

```text
1. 开始连接时
2. 断开连接时
```

---

### 6.2 不递增场景

以下场景不递增：

```text
1. 连接成功
2. 连接失败
3. 健康检测成功
4. 健康检测失败
5. 健康检测超时
6. 搜索开始
7. 搜索结束
8. 搜索超时
```

---

### 6.3 原因

开始连接时递增：

```text
使旧健康检测和旧连接请求失效。
```

断开连接时递增：

```text
使断开前的连接结果和健康检测结果失效。
```

连接成功时不递增：

```text
连接成功属于开始连接时创建的同一代状态。
```

连接失败时不递增：

```text
连接失败不应额外制造新代际；旧连接是否仍存在由 WindowManager 状态决定。
```

## 7. 搜索请求防覆盖

### 7.1 发起搜索

```python
def _start_search(self, keyword: str) -> None:
    keyword = keyword.strip()

    if not keyword:
        self._invalidate_search()
        self._clear_search_results()
        self._show_search_hint("请输入窗口标题、进程名或 PID")
        return

    self._search_request_id += 1
    request_id = self._search_request_id
    self._active_search_request_id = request_id

    self._searching = True
    self._start_search_timeout(request_id)

    self._task_runner.start_search(request_id, keyword)
```

---

### 7.2 搜索结果返回

```python
def _on_search_finished(self, request_id: int, results: list[WindowInfo]) -> None:
    self._stop_search_timeout(request_id)

    if request_id != self._active_search_request_id:
        return

    self._searching = False

    if not results:
        self._show_search_empty()
        return

    self._render_search_results(results)
```

---

### 7.3 搜索异常返回

```python
def _on_search_failed(self, request_id: int, message: str) -> None:
    self._stop_search_timeout(request_id)

    if request_id != self._active_search_request_id:
        return

    self._searching = False
    self._show_search_error(message)
```

---

### 7.4 使搜索失效

```python
def _invalidate_search(self) -> None:
    self._search_request_id += 1
    self._active_search_request_id = self._search_request_id
    self._searching = False
```

说明：

```text
通过递增并设置 active id，使所有旧搜索结果失效。
```

## 9. 搜索防抖设计

### 9.1 触发规则

## 10. 连接请求防覆盖

### 10.1 发起连接

```python
def _start_connect(self, window_info: WindowInfo) -> None:
    if self._connecting:
        return

    self._connect_request_id += 1
    request_id = self._connect_request_id
    self._active_connect_request_id = request_id

    self._connection_generation += 1
    generation = self._connection_generation
    self._connect_request_generations[request_id] = generation

    self._connecting = True

    self._stop_health_timer()
    self._health_check_running = False
    self._invalidate_health_checks()

    self._start_connect_timeout(request_id)

    options = ConnectOptions()
    self._task_runner.start_connect(
        request_id=request_id,
        window_info=window_info,
        options=options,
    )
```

---

### 10.2 连接结果返回

```python
def _on_connect_finished(self, request_id: int, result: ConnectResult) -> None:
    self._stop_connect_timeout(request_id)

    request_generation = self._connect_request_generations.pop(
        request_id,
        None,
    )

    if not self._is_connect_result_current(request_id, request_generation):
        if getattr(result, "success", False):
            self._window_manager.disconnect_window()
        return

    self._connecting = False

    if result.success:
        self._show_connect_success(result)
        self._start_health_timer()
    else:
        self._show_connect_failure(result.message)

        if self._window_manager.is_connected:
            self._start_health_timer()
```

---

### 10.3 连接异常返回

```python
def _on_connect_failed(self, request_id: int, message: str) -> None:
    self._stop_connect_timeout(request_id)

    request_generation = self._connect_request_generations.pop(
        request_id,
        None,
    )

    if not self._is_connect_result_current(request_id, request_generation):
        return

    self._connecting = False
    self._show_connect_error(message)

    if self._window_manager.is_connected:
        self._start_health_timer()
```

---

### 10.4 连接结果有效性判断

```python
def _is_connect_result_current(
    self,
    request_id: int,
    request_generation: int | None,
) -> bool:
    if request_id != self._active_connect_request_id:
        return False

    if request_generation != self._connection_generation:
        return False

    return True
```

## 11. 健康检测请求防覆盖

### 11.1 发起健康检测

```python
def _start_health_check(self) -> None:
    if self._health_check_running:
        return

    if self._connecting:
        return

    self._health_request_id += 1
    request_id = self._health_request_id
    self._active_health_request_id = request_id

    generation = self._connection_generation
    self._health_request_generations[request_id] = generation

    self._health_check_running = True
    self._start_health_timeout(request_id)

    self._task_runner.start_health_check(request_id)
```

---

### 11.2 健康检测结果返回

```python
def _on_health_finished(
    self,
    request_id: int,
    result: HealthCheckResult,
) -> None:
    self._stop_health_timeout(request_id)

    request_generation = self._health_request_generations.pop(
        request_id,
        None,
    )

    if request_id == self._active_health_request_id:
        self._health_check_running = False

    if not self._is_health_result_current(request_id, request_generation):
        return

    self._update_health_ui(result)
```

## 12. 健康检测 running 标记规则

### 12.1 为什么过期结果也要清 running

即使健康检测结果已经过期，也说明对应 Worker 已经结束。

如果不清理：

```python
self._health_check_running
```

可能一直为 `True`，导致后续健康检测无法再启动。

---

### 12.2 推荐规则

```text
只要返回的 request_id 等于当前 active_health_request_id，就清理 running。
```

即：

```python
if request_id == self._active_health_request_id:
    self._health_check_running = False
```

如果该结果已经因为 generation 不匹配而失效，仍然需要清理 running。

## 13. 超时控制总体设计

### 13.1 软超时原则

P1-03 采用软超时。

超时后：

```text
1. 标记请求失效
2. 恢复 UI 运行标记
3. 展示超时提示
4. 不强杀线程
5. Worker 后续返回时按 request_id/generation 丢弃
```

---

### 13.2 不强杀线程

不使用：

```python
thread.terminate()
thread.kill()
```

原因：

- 不安全；
- 可能造成资源泄漏；
- 可能破坏 `WindowManager` 内部状态；
- Windows API 调用可能不可中断。

---

### 13.3 超时时间

| 任务 | 超时时间 |
|---|---:|
| 搜索 | 3000ms |
| 连接 | 5000ms |
| 健康检测 | 2000ms |

---

## 14. 搜索超时

### 14.1 启动搜索超时

```python
def _start_search_timeout(self, request_id: int) -> None:
    timer = QTimer(self)
    timer.setSingleShot(True)
    timer.setInterval(3000)
    timer.timeout.connect(
        lambda rid=request_id: self._on_search_timeout(rid)
    )
    self._search_timeout_timers[request_id] = timer
    timer.start()
```

---

### 14.2 停止搜索超时


### 15.2 停止连接超时

---

### 15.3 连接超时处理

```python
def _on_connect_timeout(self, request_id: int) -> None:
    self._stop_connect_timeout(request_id)

    request_generation = self._connect_request_generations.pop(
        request_id,
        None,
    )

    if not self._is_connect_result_current(request_id, request_generation):
        return

    self._active_connect_request_id = 0
    self._connection_generation += 1
    self._connecting = False

    self._window_manager.disconnect_window()

    self._show_connect_error("连接窗口超时，请稍后重试")
```

---

### 15.4 连接超时后迟到结果

如果迟到连接结果返回：

```python
def _on_connect_finished(self, request_id: int, result: ConnectResult) -> None:
    self._stop_connect_timeout(request_id)

    request_generation = self._connect_request_generations.pop(
        request_id,
        None,
    )

    if not self._is_connect_result_current(request_id, request_generation):
        if getattr(result, "success", False):
            self._window_manager.disconnect_window()
        return

    ...
```

说明：

```text
如果迟到结果显示连接成功，但请求已失效，则再次调用 disconnect_window() 兜底清理。
```

---

### 15.5 为什么连接超时要调用 `disconnect_window()`

因为连接 Worker 可能已经部分执行，甚至可能在超时附近修改了 Core 状态。

连接超时时调用：

```python
self._window_manager.disconnect_window()
```

用于尽量恢复明确的未连接状态。

如果迟到成功结果后续返回，再次调用：

```python
disconnect_window()
```

作为二次兜底。

## 17. 断开连接时的废弃策略

### 17.1 断开连接流程

```python

    self._active_health_request_id = 0
```

## 24. 验收标准

P1-03 完成后，应满足以下验收标准。

### 24.1 请求防覆盖验收

- 连续搜索时，只展示最后一次搜索结果；
- 旧搜索结果不会覆盖新搜索结果；
- 旧搜索错误不会覆盖新搜索状态；
- 连接旧结果不会覆盖当前连接状态；
- 健康检测旧结果不会覆盖当前连接状态。

---

### 24.2 generation 验收

- 开始连接时 `connection_generation` 递增；
- 断开连接时 `connection_generation` 递增；
- 连接成功时不额外递增；
- 连接失败时不额外递增；
- 断开前发出的健康检测结果在断开后返回时被丢弃；
- 开始新连接前发出的健康检测结果被丢弃；
- 旧连接结果在 generation 不匹配时被丢弃。

---

### 24.3 防抖验收

- 输入框连续变化不会立即发起大量搜索；
- 停止输入 300ms 后发起搜索；
- 点击搜索按钮立即搜索；
- 回车立即搜索；
- 清空输入框不发 Worker；
- 清空输入框会使旧搜索结果失效。

---

### 24.4 超时验收

- 搜索超过 3000ms 显示超时；
- 连接超过 5000ms 显示超时；
- 健康检测超过 2000ms 显示超时；
- 超时不强杀线程；
- 超时后迟到结果不会更新 UI；
- 连接超时后调用 `disconnect_window()`；
- 健康检测超时不改变连接生命周期；
- 健康检测超时后下一轮检测仍可继续。

---

### 24.5 断开验收

- 断开连接会停止健康检测；
- 断开连接会废弃旧连接结果；
- 断开连接会废弃旧健康检测结果；
- 断开连接不废弃搜索结果；
- 断开连接后迟到健康检测结果不会显示“已连接”；
- 断开连接后迟到连接成功不会显示“连接成功”。

## 26. 最终结论

P1-03 的核心设计结论是：

```text
异步任务结果必须经过有效性判断后，才能更新 UI 或影响连接状态展示。
```

本阶段通过：

```text
request_id
connection_generation
300ms 搜索防抖
软超时
迟到结果废弃
```

解决 UI 异步任务中的时序一致性问题。

其中：

```text
搜索结果使用 request_id 防覆盖
连接结果使用 request_id + connection_generation 防覆盖
健康检测结果使用 request_id + connection_generation 防覆盖
```

超时采用软超时：

```text
不强杀线程，只标记请求失效并废弃迟到结果。
```

连接超时因可能影响 Core 状态，需额外调用：

```python
disconnect_window()
```

做兜底清理。

P1-03 完成后，游戏连接页可以在异步搜索、异步连接、周期健康检测同时存在的情况下，保持 UI 状态一致，不被旧请求、迟到请求或断开前请求污染。
