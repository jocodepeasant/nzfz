## 1. 文档目标

本文档用于定义游戏连接页中窗口相关 Core 操作的 UI 异步执行方案。

P0 阶段已经完成窗口连接 Core 层闭环：

```text
搜索窗口
    ↓
连接窗口
    ↓
保存连接状态
    ↓
查询状态
    ↓
断开连接
```

P1-01 已补充：

```text
WindowManager.check_health()
```

用于检测已连接窗口是否健康、是否执行就绪。

但这些 Core 接口均为同步接口：

```python
WindowManager.search_windows(keyword)
WindowManager.connect_window(window_info, options)
WindowManager.check_health()
```

如果 UI 主线程直接调用这些方法，可能导致 PySide6 界面卡顿、按钮无响应、状态刷新滞后，甚至被用户误认为程序假死。

P1-02 的目标是：

```text
在 UI 层引入异步 Worker 机制，使搜索、连接、健康检测均在后台线程执行，并通过 Qt Signal 将结果安全返回 UI 主线程。
```

---

## 2. 设计结论

根据 P1 设计讨论，P1-02 最终采用以下方案：

| 项目 | 最终选择 |
|---|---|
| Core 层是否改为异步 | 否 |
| Core 层是否依赖 Qt | 否 |
| 异步调度位置 | UI 层 |
| 线程模型 | `QThread + QObject Worker` |
| 是否使用 `QThreadPool + QRunnable` | 否 |
| 是否使用 Python `threading.Thread` | 否 |
| Worker 类型 | 三个专用 Worker |
| 搜索 Worker | `WindowSearchWorker` |
| 连接 Worker | `WindowConnectWorker` |
| 健康检测 Worker | `WindowHealthCheckWorker` |
| 是否新增任务运行器 | 是 |
| 任务运行器 | `WindowTaskRunner` |
| Worker 结果返回方式 | Qt Signal |
| Worker 信号是否携带 `request_id` | 是 |
| Worker 是否捕获异常 | 是 |
| Worker 异常是否穿透线程边界 | 否 |
| UI 是否直接更新自 Worker 线程 | 否 |
| 搜索与连接是否互斥 | 是 |
| 连接任务是否允许并发 | 否 |
| 健康检测是否允许堆积 | 否 |
| 健康检测中是否允许搜索 | 是 |
| 连接中是否暂停健康检测 | 是 |
| 超时控制 | P1-03 实现，P1-02 预留 |
| 搜索防抖 | P1-03 实现 |
| 旧结果防覆盖 | P1-03 实现，P1-02 先透传 `request_id` |
| UI 状态机 | P1-04 实现 |
| 错误文案标准化 | P1-05 实现 |

## 4. 为什么 Core 层保持同步

P1-02 明确：

```text
Core 层保持同步，异步只属于 UI 调度层。
```

原因如下。

### 4.1 保持 Core 纯净

`WindowManager` 是执行器 Core 层基础类，应避免引入：

```python
PySide6
QThread
QObject
Signal
QTimer
```

否则 Core 层会被 UI 框架绑定，不利于后续：

- 单元测试；
- CLI 调用；
- 服务化；
- 脚本调用；
- 非 UI 场景复用。

---

### 4.2 便于测试

同步接口更容易测试。

例如：

```python
result = manager.search_windows("逆战")
connect_result = manager.connect_window(info, options)
health = manager.check_health()
```

测试时无需启动 Qt 事件循环。

---

### 4.3 线程生命周期属于 UI

搜索、连接、健康检测是否异步，属于 UI 交互体验问题。

UI 层更适合决定：

- 什么时候启动任务；
- 是否禁用按钮；
- 是否跳过健康检测；
- 什么时候废弃旧结果；
- 是否显示超时；
- 什么时候停止定时器。

---

### 4.4 避免 Core 隐式并发

如果 Core 内部自行创建线程，容易造成：

- 连接状态并发修改；
- `_connected_window` 不可控变化；
- `_last_error` 竞争；
- UI 难以知道任务生命周期；
- 后续状态机难以维护。

---

## 5. 线程模型选型

### 5.1 采用方案

P1-02 采用：

```text
QThread + QObject Worker
```

典型流程：

```text
UI 主线程
    ↓
创建 Worker(QObject)
    ↓
创建 QThread
    ↓
worker.moveToThread(thread)
    ↓
thread.started.connect(worker.run)
    ↓
worker.finished / worker.failed 发回 UI
    ↓
thread.quit()
    ↓
worker.deleteLater()
    ↓
thread.deleteLater()
    ↓
清理任务引用
```

---

### 5.2 不采用 `QThreadPool + QRunnable`

原因：

- `QRunnable` 本身不是 `QObject`；
- Signal 需要额外封装；
- 对初期项目可读性不如 `QObject Worker`；
- 生命周期清理不如 `QThread + QObject` 直观；
- 当前任务数量少，不需要线程池优化。

---

### 5.3 不采用 Python `threading.Thread`

原因：

- 与 Qt 主线程通信不如 Signal/Slot 自然；
- 容易误更新 UI；
- 需要额外封装回主线程；
- 与 PySide6 项目风格不一致。

---

## 7. Worker 类设计

### 7.1 Worker 设计原则

所有 Worker 遵循以下原则：

```text
1. 继承 QObject
2. 只执行一个 Core 方法
3. 输入在构造函数中传入
4. 对外暴露 run() 槽函数
5. 成功时 emit finished
6. 异常时 emit failed
7. 信号必须携带 request_id
8. 不直接更新 UI
9. 不吞掉异常但要转换为 failed 信号
10. 不负责线程创建和销毁
```

---

### 7.2 `WindowSearchWorker`

#### 7.2.1 职责

异步执行：

```python
WindowManager.search_windows(keyword)
```

---

#### 7.2.2 输入

| 参数 | 类型 | 说明 |
|---|---|---|
| `request_id` | `int` | 搜索请求编号 |
| `window_manager` | `WindowManager` | Core 窗口管理器 |
| `keyword` | `str` | 搜索关键词 |

---

#### 7.2.3 信号

```python
finished = Signal(int, list)
failed = Signal(int, str)
```

含义：

```text
finished(request_id, results)
failed(request_id, error_message)
```

---

#### 7.2.4 执行规则

```python
results = self._window_manager.search_windows(self._keyword)
self.finished.emit(self._request_id, results)
```

如果发生未预期异常：

```python
self.failed.emit(self._request_id, f"任务执行异常：{error}")
```

---

### 7.3 `WindowConnectWorker`

#### 7.3.1 职责

异步执行：

```python

```

---

#### 7.3.2 输入

| 参数 | 类型 | 说明 |
|---|---|---|
| `request_id` | `int` | 连接请求编号 |
| `window_manager` | `WindowManager` | Core 窗口管理器 |
| `window_info` | `WindowInfo` | 待连接窗口 |
| `options` | `ConnectOptions` | 连接选项 |

---

#### 7.3.3 信号

```python

failed = Signal(int, str)
```

含义：

```text

failed(request_id, error_message)
```

其中：

```text

```

---

#### 7.3.4 执行规则


```text
即使 ConnectResult.success=False，也属于业务失败，仍然走 finished。
```


## 8. `window_workers.py` 推荐代码骨架

```python
from __future__ import annotations

from PySide6.QtCore import QObject, Signal, Slot

from nzfz_executor.core.models import (
    ConnectOptions,
    WindowInfo,
)
from nzfz_executor.core.window_manager import WindowManager


class WindowSearchWorker(QObject):
    finished = Signal(int, list)
    failed = Signal(int, str)

    def __init__(
        self,
        request_id: int,
        window_manager: WindowManager,
        keyword: str,
        parent: QObject | None = None,
    ) -> None:
        super().__init__(parent)
        self._request_id = request_id
        self._window_manager = window_manager
        self._keyword = keyword

    @Slot()
    def run(self) -> None:
        try:
            results = self._window_manager.search_windows(self._keyword)
            self.finished.emit(self._request_id, results)
        except Exception as error:
            self.failed.emit(
                self._request_id,
                f"任务执行异常：{error}",
            )


class WindowConnectWorker(QObject):
    finished = Signal(int, object)
    failed = Signal(int, str)

    def __init__(
        self,
        request_id: int,
        window_manager: WindowManager,
        window_info: WindowInfo,
        options: ConnectOptions,
        parent: QObject | None = None,
    ) -> None:
        super().__init__(parent)
        self._request_id = request_id
        self._window_manager = window_manager
        self._window_info = window_info
        self._options = options

    @Slot()
    def run(self) -> None:
        try:
            result = self._window_manager.connect_window(
                self._window_info,
                self._options,
            )
            self.finished.emit(self._request_id, result)
        except Exception as error:
            self.failed.emit(
                self._request_id,
                f"任务执行异常：{error}",
            )


class WindowHealthCheckWorker(QObject):
    finished = Signal(int, object)
    failed = Signal(int, str)

    def __init__(
        self,
        request_id: int,
        window_manager: WindowManager,
        parent: QObject | None = None,
    ) -> None:
        super().__init__(parent)
        self._request_id = request_id
        self._window_manager = window_manager

    @Slot()
    def run(self) -> None:
        try:
            result = self._window_manager.check_health()
            self.finished.emit(self._request_id, result)
        except Exception as error:
            self.failed.emit(
                self._request_id,
                f"任务执行异常：{error}",
            )
```

## 9. `WindowTaskRunner` 设计

### 9.1 职责

`WindowTaskRunner` 是 UI Worker 层的轻量任务运行器。

负责：

- 持有 `WindowManager`；
- 创建对应 Worker；
- 创建 `QThread`；
- 将 Worker 移动到线程；
- 连接线程启动信号到 Worker `run()`；
- 转发 Worker 成功/失败信号；
- 统一退出线程；
- 统一释放 Worker；
- 统一释放 QThread；
- 保存任务引用，避免提前 GC；
- 为后续 P1-03 超时和废弃结果预留扩展点。

---

### 9.2 对外信号

```python
search_finished = Signal(int, list)
search_failed = Signal(int, str)

connect_finished = Signal(int, object)
connect_failed = Signal(int, str)

health_finished = Signal(int, object)
health_failed = Signal(int, str)
```

---

### 9.3 对外方法

```python
start_search(request_id: int, keyword: str) -> None

start_connect(
    request_id: int,
    window_info: WindowInfo,
    options: ConnectOptions,
) -> None

start_health_check(request_id: int) -> None
```

---

### 9.4 内部任务存储

推荐：

```python
self._tasks: dict[str, tuple[QThread, QObject]]
```

任务 key：

```text
search:{request_id}
connect:{request_id}
health:{request_id}
```

---

### 9.5 清理规则

任务成功或失败后必须执行：

```text
1. thread.quit()
2. worker.deleteLater()
3. thread.deleteLater()
4. 从 _tasks 中移除
```

注意：

```text
必须在 thread.finished 后再从 _tasks 移除，避免对象过早释放。
```

---

## 10. `window_task_runner.py` 推荐代码骨架

```python
from __future__ import annotations

from PySide6.QtCore import QObject, QThread, Signal

from nzfz_executor.core.models import ConnectOptions, WindowInfo
from nzfz_executor.core.window_manager import WindowManager

from ui.workers.window_workers import (
    WindowConnectWorker,
    WindowHealthCheckWorker,
    WindowSearchWorker,
)


class WindowTaskRunner(QObject):
    search_finished = Signal(int, list)
    search_failed = Signal(int, str)

    connect_finished = Signal(int, object)
    connect_failed = Signal(int, str)

    health_finished = Signal(int, object)
    health_failed = Signal(int, str)

    def __init__(
        self,
        window_manager: WindowManager,
        parent: QObject | None = None,
    ) -> None:
        super().__init__(parent)
        self._window_manager = window_manager
        self._tasks: dict[str, tuple[QThread, QObject]] = {}

    def start_search(self, request_id: int, keyword: str) -> None:
        worker = WindowSearchWorker(
            request_id=request_id,
            window_manager=self._window_manager,
            keyword=keyword,
        )
        self._start_task(
            task_key=f"search:{request_id}",
            worker=worker,
            finished_signal=worker.finished,
            failed_signal=worker.failed,
            outward_finished_signal=self.search_finished,
            outward_failed_signal=self.search_failed,
        )

    def start_connect(
        self,
        request_id: int,
        window_info: WindowInfo,
        options: ConnectOptions,
    ) -> None:
        worker = WindowConnectWorker(
            request_id=request_id,
            window_manager=self._window_manager,
            window_info=window_info,
            options=options,
        )
        self._start_task(
            task_key=f"connect:{request_id}",
            worker=worker,
            finished_signal=worker.finished,
            failed_signal=worker.failed,
            outward_finished_signal=self.connect_finished,
            outward_failed_signal=self.connect_failed,
        )

    def start_health_check(self, request_id: int) -> None:
        worker = WindowHealthCheckWorker(
            request_id=request_id,
            window_manager=self._window_manager,
        )
        self._start_task(
            task_key=f"health:{request_id}",
            worker=worker,
            finished_signal=worker.finished,
            failed_signal=worker.failed,
            outward_finished_signal=self.health_finished,
            outward_failed_signal=self.health_failed,
        )

    def _start_task(
        self,
        task_key: str,
        worker: QObject,
        finished_signal,
        failed_signal,
        outward_finished_signal,
        outward_failed_signal,
    ) -> None:
        thread = QThread(self)
        worker.moveToThread(thread)

        thread.started.connect(worker.run)

        finished_signal.connect(outward_finished_signal)
        failed_signal.connect(outward_failed_signal)

        finished_signal.connect(thread.quit)
        failed_signal.connect(thread.quit)

        finished_signal.connect(worker.deleteLater)
        failed_signal.connect(worker.deleteLater)

        thread.finished.connect(thread.deleteLater)
        thread.finished.connect(lambda key=task_key: self._cleanup_task(key))

        self._tasks[task_key] = (thread, worker)
        thread.start()

    def _cleanup_task(self, task_key: str) -> None:
        self._tasks.pop(task_key, None)
```

## 11. 信号语义

### 11.1 `finished` 与 `failed` 的区别

| 信号 | 含义 |
|---|---|
| `finished` | Core 方法正常返回结果 |
| `failed` | Worker 执行过程中发生未预期异常 |

业务失败不等于 Worker 失败。

例如：

```python
ConnectResult.success == False
```

表示连接业务失败，但 `connect_window()` 正常返回，因此应走：

```text
connect_finished
```

而不是：

```text
connect_failed
```

---

### 11.2 搜索无结果

搜索无结果属于正常业务结果。

应返回：

```python
search_finished(request_id, [])
```

不应返回：

```python
search_failed(request_id, "未找到窗口")
```

---

### 11.3 健康检测不健康

健康检测返回不健康属于正常业务结果。

应返回：

```python
health_finished(request_id, health_result)
```

不应返回：

```python
health_failed(request_id, "窗口不健康")
```

只有 Worker 自身异常才走 `failed`。

## 12. 并发控制规则

### 12.1 总体原则

P1-02 不实现完整 UI 状态机，但必须明确基础并发规则。

---

### 12.2 搜索与连接互斥

规则：

```text
搜索中不允许连接。
连接中不允许搜索。
```

UI 层应在搜索中禁用连接按钮，在连接中禁用搜索按钮。

原因：

- 避免连接旧搜索结果；
- 避免连接状态和搜索结果交错；
- 降低 `_last_error` 竞争影响；
- 简化 P1-04 UI 状态机。

---

### 12.3 连接任务不允许并发

规则：

```text
同一时间最多允许一个连接任务。
```

连接中：

```text
禁用连接按钮
禁用搜索按钮
暂停健康检测
```

连接完成后：

| 连接结果 | 行为 |
|---|---|
| 成功 | 启动健康检测 |
| 失败且无旧连接 | 保持健康检测停止 |
| 失败但旧连接仍存在 | 可恢复健康检测 |

---

### 12.4 健康检测不允许堆积

规则：

```text
上一轮健康检测未完成时，跳过本轮。
```

UI 层维护：

```python
self._health_check_running: bool
```

定时器触发时：

```python
if self._health_check_running:
    return
```

Worker 返回后：

```python
self._health_check_running = False
```

---

### 12.5 健康检测中允许搜索

规则：

```text
健康检测中允许搜索。
```

原因：

- 搜索不改变连接状态；
- 用户可能在已连接状态下搜索其他窗口；
- 后续可支持切换连接目标。

---

### 12.6 健康检测中发起连接

规则：

```text
发起连接前暂停健康检测。
```

连接完成后再根据连接结果恢复或保持停止。

---

### 12.7 断开连接

`disconnect_window()` 本身很快，P1-02 不要求异步执行。

但断开连接时，UI 应：

```text
停止健康检测 timer
标记当前健康检测不再有效
清空或更新连接状态展示
```

“废弃断开前已发出的健康检测结果”属于 P1-03 的 request generation 机制。

---

## 13. 与健康检测定时器的关系

P1-02 不负责创建完整健康检测状态机，但需定义和定时器的关系。

推荐 UI 层：

```text
连接成功后启动 QTimer
定时器间隔 2000ms
每次 timeout 调用 WindowTaskRunner.start_health_check()
```

伪代码：

```python
def _start_health_timer(self) -> None:
    self._health_timer.start(2000)

def _stop_health_timer(self) -> None:
    self._health_timer.stop()
    self._health_check_running = False

def _on_health_timer_timeout(self) -> None:
    if self._health_check_running:
        return

    self._health_check_running = True
    self._health_request_id += 1

    self._task_runner.start_health_check(
        request_id=self._health_request_id,
    )
```

Worker 完成：

```python
def _on_health_finished(self, request_id: int, result: HealthCheckResult) -> None:
    self._health_check_running = False
    # P1-03 判断 request_id 是否有效
    # P1-04 更新 UI 状态机
```

Worker 异常：

```python
def _on_health_failed(self, request_id: int, message: str) -> None:
    self._health_check_running = False
    # 展示或记录错误
```

## 15. 线程安全约束

### 15.1 UI 更新必须在主线程

Worker 不允许直接访问或更新：

- `QTableWidget`
- `QPushButton`
- `QLabel`
- `QLineEdit`
- 其他 QWidget

UI 更新只能在主线程 slot 中进行。

---

### 15.2 `WindowManager` 共享实例风险

`WindowTaskRunner` 持有同一个 `WindowManager` 实例，三个 Worker 都可能调用它。

因此需要通过 UI 层并发规则降低风险：

| 操作组合 | 是否允许 | 说明 |
|---|---:|---|
| 搜索 + 搜索 | 可连续发起，P1-03 处理旧结果 |
| 搜索 + 连接 | 不允许 |
| 连接 + 连接 | 不允许 |
| 连接 + 健康检测 | 不允许，连接前暂停健康检测 |
| 健康检测 + 健康检测 | 不允许堆积 |
| 健康检测 + 搜索 | 允许 |

---

### 15.3 不强制终止线程

P1-02 不实现：

```text
强制终止正在执行的 Windows API 调用
```

原因：

- Python/Qt 线程强杀不安全；
- Windows API 可能处于不可中断状态；
- 强杀可能导致资源泄露或状态不一致。

后续 P1-03 通过：

```text
超时后废弃结果
```

来保障 UI 体验，而不是强杀线程。

---

## 16. 错误处理规则

### 16.1 Worker 必须捕获异常

所有 Worker 的 `run()` 必须：

```python
try:
    ...
except Exception as error:
    self.failed.emit(self._request_id, f"任务执行异常：{error}")
```

---

### 16.2 `failed` 只表示未预期异常

以下场景不应走 `failed`：

| 场景 | 应走信号 |
|---|---|
| 搜索无结果 | `search_finished` |
| 非 Windows 搜索返回空 | `search_finished` |
| 连接被业务校验拒绝 | `connect_finished` |
| 连接最小化窗口失败 | `connect_finished` |
| 健康检测返回不健康 | `health_finished` |
| 健康检测返回未连接 | `health_finished` |

以下场景可走 `failed`：

| 场景 | 应走信号 |
|---|---|
| Worker 代码异常 | `failed` |
| 未预期 AttributeError | `failed` |
| 未预期 TypeError | `failed` |
| Core 方法抛出未捕获异常 | `failed` |

---

### 16.3 错误文案

P1-02 临时使用：

```text
任务执行异常：{error}
```

最终文案由 P1-05 标准化。

P1-03 实现：

## 21. 验收标准

P1-02 完成后，应满足以下验收标准。

### 21.1 架构验收

- `WindowManager` 不引入 PySide6；
- `WindowManager` 不创建线程；
- `WindowManager` 不管理 QTimer；
- UI Worker 层负责异步执行；
- Worker 和 TaskRunner 放在 `ui/workers/` 下；
- Core 层同步接口不被破坏。

---

### 21.2 功能验收

- 搜索窗口异步执行；
- 连接窗口异步执行；
- 健康检测异步执行；
- Worker 结果通过 Signal 返回 UI；
- Worker 异常通过 failed Signal 返回 UI；
- 所有 Worker 信号携带 `request_id`；
- 搜索无结果不视为 Worker 失败；
- 连接业务失败不视为 Worker 失败；
- 健康检测不健康不视为 Worker 失败。

---

### 21.3 UI 响应验收

- 搜索过程中 UI 不假死；
- 连接过程中 UI 不假死；
- 健康检测过程中 UI 不假死；
- 用户可以拖动窗口、切换页签、查看状态；
- 后台任务完成后 UI 能正确接收结果。

---

### 21.4 生命周期验收

- Worker 不会被提前回收；
- QThread 不会被提前回收；
- 任务完成后 Worker 释放；
- 任务完成后 QThread 释放；
- 任务完成后 `_tasks` 清理；
- 多次搜索不会导致线程对象持续堆积；
- 多次健康检测不会导致线程对象持续堆积。

---

### 21.5 并发验收

- 搜索中不能发起连接；
- 连接中不能发起搜索；
- 同一时间不能发起多个连接任务；
- 健康检测不会堆积；
- 连接中健康检测暂停；
- 健康检测中允许搜索。

## 23. 最终结论

P1-02 的核心设计结论是：

```text
Core 层保持同步，UI 层负责异步调度。
```

本阶段通过：

```text
QThread + QObject Worker
```

实现三个专用异步任务：

```text
WindowSearchWorker
WindowConnectWorker
WindowHealthCheckWorker
```

并通过：

```text
WindowTaskRunner
```

统一管理线程生命周期、信号转发和任务清理。

P1-02 完成后，以下操作均不会阻塞 UI 主线程：

```text
窗口搜索
窗口连接
健康检测
```

同时为后续 P1-03 的：

```text
请求防覆盖
搜索防抖
超时控制
断开后废弃旧结果
```

提供稳定基础。
