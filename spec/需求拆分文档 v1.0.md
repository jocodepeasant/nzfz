# 《逆战：未来》塔防自动化执行器 · 需求拆分与优先级规划文档

## 1. 文档目的

本文档面向当前项目中的 **自动化执行器** 子系统，对已有窗口连接需求进行进一步拆分，形成可执行、可排期、可验收的需求规划。

当前项目整体由两个子系统组成：

```text
地图配置器 → script.json → 自动化执行器 → 游戏
```

其中：

- 地图配置器已独立负责地图、陷阱、区域、槽位、波次等配置；
- 脚本 JSON 已由共享 Schema 约束；
- 本文档只关注 **自动化执行器**；
- 本次重点拆分的是执行器中的 **游戏窗口连接能力**；
- 不修改地图配置器；
- 不修改既有 JSON Schema；
- 不随意扩展当前版本范围。


---

## 本次修订：支持搜索最小化窗口但禁止连接

本次修订统一三份文档中的窗口搜索与连接边界：

- **搜索阶段**：允许搜索并返回最小化窗口。
- **搜索结果**：必须通过 `WindowInfo.is_minimized` 标记窗口是否最小化。
- **搜索过滤**：不可见窗口仍默认过滤；最小化窗口不应因 `GetWindowRect` 返回异常占位尺寸而被过滤。
- **连接阶段**：仍然禁止连接最小化窗口，即使 `ConnectOptions.restore_if_minimized=True` 也不自动恢复。
- **UI 行为**：可以展示最小化窗口，但连接按钮或连接结果必须明确提示“窗口已最小化，请恢复窗口后重试”。

## 2. 当前执行器在系统中的定位

自动化执行器是整个工具链中的运行端，负责：

```text
读取脚本 JSON → 连接游戏窗口 → 根据窗口上下文执行自动化动作
```

当前阶段的关键不是直接实现完整通关自动化，而是先建立执行器的基础能力：

```text
稳定识别并绑定目标游戏窗口
```

因为后续所有执行动作都依赖这个窗口上下文。

例如：

- 截图需要窗口客户区；
- 坐标换算需要客户区矩形；
- 鼠标点击需要目标窗口位置；
- 键盘输入需要目标窗口激活状态；
- 波次执行需要持续确认窗口仍然有效；
- 异常恢复需要健康检测状态。

因此，窗口连接能力是执行器的底座能力，优先级高于截图、输入、动作执行、图像识别等高级能力。

---

## 3. 需求拆分原则

本次需求拆分遵循以下原则：

### 3.1 按执行链路拆分

执行器窗口连接能力的完整链路为：

```text
数据模型 → 窗口搜索 → 连接校验 → 激活窗口 → 保存连接状态 → 健康检测 → UI 状态管理 → 与执行器主流程集成
```

每个阶段应能够独立开发、独立测试、独立验收。

---

### 3.2 按风险优先级拆分

优先实现高风险、高依赖能力，例如：

- Windows API 调用；
- `hwnd` 获取；
- 进程信息获取；
- 激活窗口；
- 健康检测；
- UI 线程隔离；
- 超时与旧请求结果防覆盖。

这些能力一旦不稳定，后续自动化执行无法可靠运行。

---

### 3.3 按依赖关系拆分

低层能力优先，高层能力后置：

```text
模型层 → Core 层 → GUI 层 → 执行器主流程集成
```

不应先做 UI 动画或脚本执行入口，再反补窗口连接核心逻辑。

---

### 3.4 按版本边界拆分

本阶段只实现窗口连接能力，不提前实现：

- 截图；
- 图像识别；
- 自动点击；
- 后台输入；
- 动作执行器；
- 自动通关逻辑；
- 自动重连策略；
- 多窗口会话管理。

---

## 4. 总体需求优先级分级

本文档将需求划分为 5 个优先级：

| 优先级 | 含义                               | 处理策略           |
| ------ | ---------------------------------- | ------------------ |
| P0     | 必须实现，否则执行器窗口连接不可用 | 当前版本必须完成   |
| P1     | 强依赖能力，影响稳定性和用户体验   | 当前版本应完成     |
| P2     | 与执行器主流程集成相关             | 当前版本建议完成   |
| P3     | 后续扩展预留                       | 只做接口或结构预留 |
| P4     | 当前明确不做                       | 不进入当前开发范围 |

```text

```

---

### 依赖关系

```text

```

### 实现要点

使用：

```python

win32gui.IsWindowVisible
win32gui.GetWindowText
win32gui.GetWindowRect
win32process.GetWindowThreadProcessId
psutil.Process(pid).name()
```

---

### 基础过滤规则

搜索时跳过以下窗口：

| 条件                       | 处理                                                         |
| -------------------------- | ------------------------------------------------------------ |
| 非有效窗口                 | 跳过                                                         |
| 不可见窗口                 | 跳过                                                         |
| 无标题窗口                 | 跳过                                                         |
| 非最小化窗口宽度小于 `200` | 跳过                                                         |
| 非最小化窗口高度小于 `150` | 跳过                                                         |
| 最小化窗口                 | 搜索阶段保留，并设置 `is_minimized=True`；不因尺寸异常被过滤 |
| 获取不到 PID               | 跳过                                                         |
| 获取不到进程名             | 跳过                                                         |

---

### 匹配度规则

```python
def calculate_match_score(keyword: str, title: str, process_name: str, pid: int) -> float:
    keyword = keyword.lower().strip()
    title_lower = title.lower()
    process_lower = process_name.lower()

    if keyword.isdigit() and int(keyword) == pid:
        return 1.0

    if title_lower == keyword:
        return 1.0

    if process_lower == keyword:
        return 0.95

    if keyword in title_lower:
        return 0.85

    if keyword in process_lower:
        return 0.8

    return 0.0
```

- [ ] 搜索结果包含 `hwnd`

## 6.4 P0-04：连接前窗口有效性校验

### 需求描述

连接窗口时不得直接信任搜索结果，必须重新校验窗口当前状态。

原因：

```text
搜索结果只是某一时刻的快照。
用户点击连接时，目标窗口可能已关闭、隐藏、最小化或 PID 已变化。
```

---

### 需求优先级

```text
P0
```

---

### 依赖关系

依赖：

```text
P0-01 数据模型
P0-02 WindowManager 基础类
P0-03 真实 Windows 窗口搜索
```

---

### 校验内容

连接时必须按顺序校验：

```text
1. hwnd 是否有效
2. 进程是否存活
3. hwnd 当前 PID 是否仍然等于 WindowInfo.pid
4. 窗口是否可见
5. 窗口是否最小化
6. 窗口尺寸是否有效
```

---

### 失败消息

| 失败场景   | 错误消息                         |
| ---------- | -------------------------------- |
| 句柄无效   | `窗口句柄已失效`                 |
| 进程退出   | `进程已退出`                     |
| PID 变化   | `窗口所属进程已变化`             |
| 窗口不可见 | `窗口不可见`                     |
| 窗口最小化 | `窗口已最小化，请恢复窗口后重试` |
| 尺寸异常   | `窗口尺寸异常`                   |

---

### 验收标准

- [ ] 目标窗口关闭后连接失败
- [ ] 目标进程退出后连接失败
- [ ] 目标窗口隐藏后连接失败
- [ ] 目标窗口最小化后连接失败
- [ ] PID 不一致时连接失败
- [ ] 尺寸异常时连接失败
- [ ] 每种失败均返回明确 `error_message`

## 6.6 P0-06：连接成功前激活目标窗口

### 需求描述

连接成功前，必须尝试激活目标游戏窗口。

如果激活失败，默认认为连接失败。

---

### 需求优先级

```text
P0
```

---

### 依赖关系

依赖：

```text
P0-04 连接前窗口有效性校验
P0-05 窗口矩形与客户区矩形获取
```

---

### 激活流程

```text
1. 如果窗口最小化，直接失败，不自动恢复
2. 调用 SetForegroundWindow(hwnd)
3. 短暂等待
4. 调用 GetForegroundWindow()
5. 如果前台窗口等于目标 hwnd，则激活成功
6. 否则激活失败
```

---

### 失败消息

```text
无法激活窗口，请尝试手动点击游戏窗口或以管理员身份运行工具
```

---

### 验收标准

- [ ] 连接成功后目标窗口成为前台窗口
- [ ] 激活失败时连接失败
- [ ] 激活失败时返回明确错误消息
- [ ] 最小化窗口不会被自动恢复
- [ ] 最小化窗口不会被连接成功

```text
1. 当前是否有连接窗口
```

```python
def get_selected_window(self) -> WindowInfo | None:
    row = self.result_table.currentRow()
    if row < 0:
        return None

    item = self.result_table.item(row, 0)
    if item is None:
        return None

    window = item.data(Qt.UserRole)
    if not isinstance(window, WindowInfo):
        return None

    return window
```

# 14. 风险分析与控制策略

## 14.1 Windows 激活失败风险

### 风险

`SetForegroundWindow` 可能失败，尤其在以下场景：

- 游戏以管理员权限运行；
- 执行器不是管理员；
- Windows 前台窗口限制；
- 游戏全屏或特殊渲染模式；
- 安全软件限制。

### 控制策略

当前版本不实现复杂激活策略，只提示：

```text
无法激活窗口，请尝试手动点击游戏窗口或以管理员身份运行工具
```

---

## 14.2 权限不足风险

### 风险

权限不足可能导致：

- 无法激活窗口；
- 无法获取窗口信息；
- 后续无法输入；
- 后续截图失败。

### 控制策略

当前版本不自动提权，只提示用户以管理员身份运行执行器。

---

## 14.3 线程竞态风险

### 风险

连接线程可能晚于超时返回，导致旧结果覆盖新状态。

### 控制策略

必须实现：

```text
连接请求 ID
```

所有连接结果返回时先校验 request_id。

---

## 14.4 健康检测误清理风险

### 风险

如果窗口最小化时直接清空连接状态，则用户恢复窗口后无法自动恢复。

### 控制策略

健康检测异常时不应轻易清空 `_connected_window`。

尤其是：

```text
WINDOW_MINIMIZED
WINDOW_HIDDEN
WINDOW_SIZE_INVALID
```

应允许后续检测恢复。

---

## 14.5 搜索异常导致整体失败风险

### 风险

枚举窗口时某个窗口获取标题、PID 或进程名失败，可能导致整个搜索失败。

### 控制策略

枚举每个窗口时独立捕获异常，单个窗口失败只跳过该窗口。

# 16. 最终结论

本次执行器窗口连接需求应按照以下顺序推进：

```text
先完成 Core 能力，再接 GUI，再做健康检测，最后接入执行器主流程。
```

核心优先级结论如下：

| 优先级 | 目标                                       |
| ------ | ------------------------------------------ |
| P0     | 形成真实窗口连接最小闭环                   |
| P1     | 形成稳定可用的 GUI 连接体验                |
| P2     | 让窗口连接成为执行器自动化主流程的前置条件 |
| P3     | 为后续输入、截图、动作执行预留结构         |
| P4     | 明确当前不做，防止需求膨胀                 |

当前版本最重要的交付结果是：

```text
执行器能够在 Windows 上搜索、连接、激活并持续监控一个真实游戏窗口。
```

该能力完成后，后续脚本执行、截图识别、坐标换算、输入控制、自动通关流程才能建立在稳定基础之上。
